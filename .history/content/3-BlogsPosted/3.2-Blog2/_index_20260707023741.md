Vì sao và cách migrate sang AWS Network Firewall gắn trực tiếp với Transit Gateway (TGW-attached)

Nếu bạn đang vận hành hạ tầng AWS theo mô hình nhiều VPC (spoke VPCs) và dùng AWS Transit Gateway (TGW) để định tuyến tập trung, thì khả năng cao bạn cũng từng (hoặc sắp) gặp bài toán: làm sao inspect traffic tập trung mà không phải dựng cả một “inspection VPC” rườm rà?

AWS Network Firewall giờ hỗ trợ native attachment vào Transit Gateway. Nói đơn giản, firewall có thể gắn trực tiếp vào TGW như một “network function attachment”, giúp bạn bỏ hẳn inspection VPC trong nhiều kiến trúc truyền thống.

Trong bài blog này, mình sẽ tóm tắt và phân tích:
- TGW-attached Network Firewall là gì?
- Vì sao nên migrate?
- Cần chuẩn bị gì?
- Migration diễn ra như thế nào (2 kiến trúc phổ biến)
- Best practices để giảm downtime và tránh sự cố routing

1. Bối cảnh cũ: Centralized inspection VPC (đúng nhưng… hơi nặng)

Trước đây, nếu bạn muốn kiểm tra/giám sát traffic tập trung (đặc biệt là egress internet), mô hình phổ biến là:
- Dựng một inspection VPC
- Tạo các firewall subnets và đặt Network Firewall endpoints ở đó
- Dùng TGW routing để “bẻ lái” traffic từ các spoke VPC qua inspection VPC rồi mới đi tiếp

Mô hình này chạy ổn, nhưng đi kèm:
- Nhiều routing table / subnet / VPC phải quản lý
- Phức tạp khi scale multi-AZ
- Khó “chargeback/allocate cost” chi tiết phí Network Firewall cho từng account/team

2. Transit Gateway-attached Network Firewall là gì?

Với native attachment, bạn không cần tạo inspection VPC nữa.

Thay vào đó:
- Bạn tạo Network Firewall
- Chỉ định Transit Gateway ID mà bạn muốn attach vào
- AWS sẽ deploy firewall endpoints vào một AWS-managed VPC (bạn không sở hữu và không quản lý VPC này)

Từ góc nhìn của bạn, firewall xuất hiện như:
- Một Transit Gateway network function attachment.
- Bạn route traffic tới nó giống như các TGW attachments khác.

Ý nghĩa lớn nhất: giảm một tầng kiến trúc và giảm khối lượng vận hành.

3. Vì sao nên migrate? (2 lợi ích nổi bật)

AWS nêu 2 lý do “đáng tiền” nhất:
(1) Flexible cost allocation (chargeback rõ ràng hơn)
Với native attachment, bạn có thể dùng Transit Gateway metering policies để phân bổ chi phí theo lượng traffic đi qua firewall.

Điểm quan trọng:
- Trước đây (không native attach), bạn chỉ allocate được TGW data processing charges.
- Còn Network Firewall charges thì không linh hoạt bằng.
- Native attachment mở ra khả năng chargeback đúng “ai xài thì trả”.

(2) Giảm độ phức tạp kiến trúc
- Bỏ inspection VPC
- Bớt routing tables/subnets cần maintain
- Tập trung quản trị network security ở một chỗ, nhưng nhẹ hơn

4. Chuẩn bị trước khi migrate: checklist ngắn gọn

Prerequisites
- Transit Gateway ID (TGW mà firewall sẽ attach vào)
- Logging configuration mới (CloudWatch log groups mới, tách log old/new trong giai đoạn chạy song song)
- Firewall policy mới (không reuse policy cũ ngay, để chỉnh policy cho firewall mới mà không ảnh hưởng production)

Key considerations (các “bẫy” cần để ý)
- Transit Gateway encryption: nếu bạn đang dùng encryption và bắt buộc phải có, thì native attachment hiện chưa hỗ trợ → bạn cần giữ kiến trúc cũ.
- NAT Gateway Elastic IPs (EIP): nếu bạn phải giữ nguyên public IP (đối tác allowlist), cần lập kế hoạch preserve EIP (phần này bài viết nói rất kỹ).
- Maintenance window: sẽ có brief outage ở một vài bước (swap route table association, replace NAT gateways), nên hãy đặt lịch bảo trì rõ ràng.

5. Chiến lược migrate “đúng bài”: chạy song song, test trước, cutover theo pha

Nguyên tắc AWS nhấn mạnh:
- Không đụng cấu hình firewall cũ lúc đầu
- Dựng firewall mới song song
- Test kỹ với 1 spoke VPC trước
- Migrate dần hoặc cutover hàng loạt khi tự tin

Điều này giúp bạn có đường lui (rollback) và giảm rủi ro.

6. Migration theo 2 kiến trúc phổ biến

Architecture 1: Inspection VPC riêng + Egress VPC riêng
Bạn có:
- Inspection VPC: đặt firewall endpoints
- Egress VPC: đặt NAT gateways

High-level steps:
1. Deploy egress VPC mới với NAT gateway “tạm”
2. Tạo Network Firewall mới và attach native vào TGW
3. Tạo 3 TGW route tables mới:
- inspection route table (associate với firewall attachment)
- egress route table (associate với egress VPC mới)
- migrated spoke route table (tạm thời để test từng spoke)
4. Test bằng cách chuyển 1 spoke VPC sang đường mới
5. Migrate các spoke còn lại (incremental hoặc đổi default route để chuyển hàng loạt)
6. (Optional) Preserve NAT EIPs bằng cách re-route về egress VPC cũ
7. Decommission tài nguyên cũ khi xác nhận ổn

Architecture 2: Inspection + Egress chung một VPC
Bạn có 1 VPC chứa cả:
- Firewall endpoints
- NAT gateways

Migration steps gần giống Architecture 1, nhưng phần preserve EIP thường “đau” hơn:
Thường phải xóa NAT gateway cũ để giải phóng EIP, rồi tạo NAT gateway mới ở egress VPC mới và “gắn lại” EIP cũ. Cần maintenance window theo từng AZ (vì NAT gateway provisioning có thời gian)

7. Một mẹo kiểm tra cực quan trọng: phát hiện asymmetric routing

AWS đề xuất kiểm chứng bằng log:
- Khi bạn chuyển 1 spoke VPC sang đường mới, hãy kiểm tra firewall alert logs:
- Nếu log có Layer 7 (application layer) details, điều đó cho thấy firewall nhìn thấy cả hai chiều traffic → routing symmetric.
- Nếu asymmetric routing, firewall chỉ thấy một chiều → stateful inspection sẽ sai và có thể drop traffic.

Đây là một “tín hiệu” rất thực dụng để tự tin trước khi cutover nhiều spoke.

8. Best practices để giảm downtime & tránh “toang” east-west traffic

AWS nhấn mạnh 3 điều:
1. Test trong dev/test account trước.
 - Migration guide repo có sẵn template Terraform/CloudFormation dựng môi trường “giống thật” để diễn tập.
2. Migrate theo pha
 - Bắt đầu với spoke không critical.
 - Monitor kỹ với owner ứng dụng.
3. Cẩn thận east-west traffic giữa spokes.
- Nếu 2 spoke nằm trên 2 “firewall path” khác nhau trong giai đoạn migrate, east-west traffic có thể đi qua 2 stateful firewalls khác nhau.
- Do state tracking độc lập, return traffic có thể bị xem là “untracked” và bị drop.
- Vì vậy: migrate các spoke có east-west traffic cùng lúc nếu có thể.

Và luôn luôn:
- Lưu cấu hình routing hiện tại trước khi đổi.
- Giữ firewall cũ hoạt động đến khi mọi thứ ổn.
- Chuẩn bị rollback plan rõ ràng.

Kết luận:
Transit Gateway-attached Network Firewall là một bước tiến theo hướng:
- Đơn giản hóa kiến trúc
- Dễ quản trị hơn
- Chargeback chi phí rõ hơn
- AWS “gánh” phần endpoint + managed VPC thay bạn

Quan trọng là migration cần làm đúng chiến lược: chạy song song, test kỹ, migrate theo pha, chú ý asymmetric routing và east-west traffic.
