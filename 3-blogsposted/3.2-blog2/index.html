Why and how to migrate to an AWS Network Firewall directly attached to a Transit Gateway (TGW-attached)

If you’re running a multi-VPC (spoked VPC) AWS infrastructure and using AWS Transit Gateway (TGW) for centralized routing, chances are you’ve encountered (or are about to encounter) this problem: how to inspect traffic centrally without having to set up a cumbersome “inspection VPC”?

AWS Network Firewall now supports native attachment to Transit Gateway. Simply put, the firewall can be directly attached to the Transit Gateway as a “network function attachment,” eliminating VPC inspection in many traditional architectures.

In this blog post, I will summarize and analyze:

  • What is a TGW-attached Network Firewall?
  • Why should you migrate?
  • What preparations are needed?
  • How does migration occur (2 common architectural styles)?
  • Best practices to reduce downtime and avoid routing issues

1. Old context: Centralized inspection VPC (correct, but… a bit heavy)

Previously, if you wanted to centrally monitor/check traffic (especially egress internet), the common model was:

  • Create a VPC inspection
  • Create firewall subnets and place Network Firewall endpoints there.
  • Use TGW routing to redirect traffic from spoke VPCs through inspection VPCs before continuing on.

This model runs well, but it comes with:

  • Multiple routing tables/subnets/VPCs to manage.
  • Multi-AZ scaling is complicated.
  • Difficult to provide a detailed “chargeback/allocate cost” for Network Firewall fees for each account/team.

2. What is a Transit Gateway-attached Network Firewall?

With native attachment, you no longer need to create an inspection VPC.

Instead:

  • You create a Network Firewall
  • Specify the Transit Gateway ID you want to attach.

AWS will deploy firewall endpoints to an AWS-managed VPC (you do not own or manage this VPC). From your perspective, a firewall appears as follows:

  • A Transit Gateway network function attachment.
  • You route traffic to it just like you would to other TGW attachments.

The biggest benefit: reducing one building level and decreasing operational workload.

3. Why should you migrate? (2 outstanding benefits)

AWS lists two of the most “worthwhile” reasons:

(1) Flexible cost allocation (more obvious chargeback)

With native attachments, you can use Transit Gateway metering policies to allocate costs based on the amount of traffic passing through the firewall.

Key point: Previously (without native attach), you could only allocate TGW data processing charges. Network firewall charges, on the other hand, are less flexible. Native attachment opens up the possibility of true chargeback: “whoever uses it pays.”

(2) Reduce architectural complexity

  • Remove VPC inspection
  • Fewer routing tables/subnets that need to be maintained
  • Centralize network security management in one place, but with a lighter workload.

4. Preparing before migrating: a concise checklist

Before going into production, AWS recommends the following preparations:

Prerequisites

  • Transit Gateway ID (the TGW that the firewall will attach to)
  • New logging configuration (new CloudWatch log groups, separating old/new logs during parallel execution)
  • New firewall policy (do not reuse the old policy immediately, so you can adjust the policy for the new firewall without affecting production)

Key considerations (pitfalls to watch out for)

  • Transit Gateway encryption: If you are using encryption and it is required, native attachments are not currently supported → you need to keep the old architecture.
  • NAT Gateway Elastic IPs (EIPs): If you need to keep your public IP (partner allowlist), you need to plan to preserve the EIP (this is discussed in detail in the article).
  • Maintenance window: There will be brief outages at some steps (swapping route table associations, replacing NAT gateways), so schedule maintenance clearly.

5. The “proper” migration strategy: parallel execution, pre-testing, phase-based cutover.

AWS principles emphasize:

  • Do not change the old firewall configuration at the beginning.
  • Set up a new firewall in parallel.
  • Test thoroughly with one VPC speaker first.
  • Gradually increase your gain or make a massive cutover when you feel confident.

This gives you a rollback option and reduces risk.

6. Migration follows two common architectural styles.

The article is divided into two architectures:

Architecture 1: Inspection of a separate VPC + Egress of a separate VPC

You have:

  • Inspection VPC: set firewall endpoints
  • Egress VPC: set NAT gateways

High-level steps:

  1. Deploy the new egress VPC with a “temporary” NAT gateway.
  2. Create a new Network Firewall and attach it natively to TGW.
  3. Create 3 new TGW route tables:
  • inspection route table (associate with firewall attachment)
  • egress route table (associate with the new egress VPC)
  • Migrated spoken route table (temporary for testing each speaker)
  1. Test by switching one VPC speaker to a new line.
  2. Migrate the remaining spokes (incremental or change the default route to transfer in bulk).
  3. (Optional) Preserve NAT EIPs by re-route to the old egress VPC
  4. Decommission old resources once confirmed to be working.

One advantage of Architecture 1 is that the EIP preservation part can be “lightweight”: because the old egress VPC is independent, you can simply change the routing association to revert to the old NAT gateways (keeping the IP) quite quickly.

Architecture 2: Inspection and Egress share a single VPC.

You have a VPC containing both:

  • Firewall endpoints
  • NAT gateways

Migration steps are similar to Architecture 1, but preserving the EIP is often more challenging:

Typically, the old NAT gateway needs to be deleted to free up EIP, then a new NAT gateway needs to be created in the new egress VPC, and the old EIP needs to be “reattached.” Maintenance windows are required for each AZ (because NAT gateway provisioning has a time limit).

7. A crucial troubleshooting tip: detect asymmetric routing.

AWS recommends verification using logs:

When you switch a spoken VPC to a new line, check the firewall alert logs:

  • If the log contains Layer 7 (application layer) details, it indicates that the firewall sees traffic in both directions → symmetric routing.
  • If asymmetric routing is used, the firewall only sees one direction → stateful inspection will be inaccurate and may drop traffic.

This is a very practical “signal” to boost confidence before cutting over many spokes.

8. Best practices to reduce downtime and avoid disruptions in east-west traffic.

AWS emphasizes three things:

  1. Test in dev/test account first.
  • The Migration guide repository includes Terraform/CloudFormation templates for creating realistic environments for drills.
  1. Phase Migration
  • Start with non-critical spokes.
  • Monitor closely with the application owner.
  1. Be careful of east-west traffic between the speakers.
  • If the two spokes are on two different firewall paths during the migration phase, east-west traffic may pass through two different stateful firewalls.
  • Due to independent state tracking, return traffic may be considered “untracked” and dropped.

Therefore: migrate to speakers with east-west traffic at the same time if possible. And always:

  • Save the current routing configuration before making changes.
  • Keep the old firewall running until everything is stable.
  • Prepare a clear rollback plan.

Conclude:

Transit Gateway-attached Network Firewall is a step forward in the following direction:

  • Simplify the architecture
  • Easier to manage
  • Chargeback provides clearer cost information.
  • AWS handles the endpoint and managed VPC for you.

The key is to execute the migration strategy correctly: run in parallel, test thoroughly, migrate in phases, and pay attention to asymmetric routing and east-west traffic.
