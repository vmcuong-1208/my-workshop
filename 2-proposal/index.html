Proposal

AI Personal Learning

A Serverless AWS Platform for Reflective Learning and AI-Assisted Progress Review

1. Executive Summary

The project involves building a web platform that helps learners keep a daily learning journal, including text and attached images. AI is then used to analyze, synthesize, and generate weekly/monthly reports on learning progress, strengths and weaknesses, study habits, and suggestions for improvement. It also includes creating review quizzes, searching, and tracking progress.

The entire system is deployed on AWS using a serverless architecture. The frontend is deployed using AWS Amplify Hosting, distributed via Amazon CloudFront, and protected by AWS WAF. User authentication is done using Amazon Cognito. The backend utilizes Amazon API Gateway, AWS Lambda, Amazon DynamoDB, Amazon S3, Amazon SQS, DLQ, Amazon CloudWatch, Amazon SNS, AWS Budgets, and Groq API for AI processing.

The goal is to create a useful learning support product that demonstrates the ability to design a modern, secure, cost-effective cloud architecture, handle asynchronous operations, and is easy to operate for small teams.

2. Project Objectives

Main Objectives:

2.1. Support for Self-Reflection Learning

Users can keep a learning journal (text + images), which is analyzed and summarized by AI into weekly/monthly reports, helping them review their learning journey in a structured way.

2.2. Personalized Learning Suggestions

AI uses journal content and progress to suggest: which lessons to focus on, which skills, and which habits need adjustment.

2.3. Serverless Architecture Practice on AWS

Applying serverless, event-driven, and AI (Groq) architecture in a real-world system, meeting the principles of AWS Well Architected (cost, security, availability, operation).

2.4. Cost Optimization for Small Teams

Designing infrastructure to run at low cost: using Lambda, S3, DynamoDB/SQS, pay-per-use, and Monitoring & Cost Guardrails (CloudWatch + Budgets).

3. Problem Statement

Current Situation

Learning notes are often scattered across Notion, Google Docs, screenshots, chat messages, and AI conversations. This makes it difficult for learners to review their progress over time. Mentors also need to read many separate logs if they want to understand a learner’s development, repeated mistakes, and improvement trend.

General AI chat tools can summarize text, but they usually do not understand the learner’s structured timeline, tags, tasks, categories, or historical progress. As a result, feedback can become generic and hard to turn into concrete actions.

Proposed Solution

The platform centralizes learning records in one application. Each journal entry contains structured fields such as title, content, date, category, tags, commands, errors, solutions, mood, difficulty, study time, and image attachments.

AI features are added on top of this structured data:

  • Weekly or monthly learning reports.
  • Strengths, weaknesses, and recommendations.
  • Quiz generation for review.
  • Search across personal learning logs.
  • Analytics summary by category, tag, mood, difficulty, and study time.

Long-running AI tasks are processed asynchronously through SQS and a Lambda AI Worker, so the frontend remains responsive and users can poll for completion.

4. Solution Architecture

Edge and Frontend Layer

  • AWS Amplify Hosting deploys the React/Vite SPA directly from GitHub.
  • Amazon CloudFront delivers static assets globally over HTTPS.
  • AWS WAF protects the public edge with managed rules and rate-based controls.

Authentication Layer

  • Amazon Cognito User Pool handles sign-up, sign-in, and optional Google federation.
  • The SPA receives JWT tokens from Cognito.
  • API Gateway JWT Authorizer protects user-specific API routes.

Application Layer

  • Amazon API Gateway HTTP API exposes routes such as /health, /me, /logs, /ai/reports, /quiz, /search, and /analytics/summary.
  • Lambda API (AI-Personal-Learning-Handle) handles synchronous requests, journal CRUD, image presigned URL generation, and SQS job submission.
  • Lambda Search Handler supports keyword, topic, tag, and date filtering.
  • Lambda Analytics Handler aggregates journal data into dashboard metrics.

Data and File Layer

  • DynamoDB stores journal logs, AI reports, quizzes, and quiz attempts.
  • S3 stores user-uploaded learning images.
  • Images are uploaded through presigned URLs so they do not pass through Lambda or API Gateway.

AI Processing Layer

  • SQS stores AI report and quiz jobs.
  • DLQ stores failed messages for troubleshooting.
  • Lambda AI Worker (AI-Personal-Learning-Worker) reads jobs from SQS, loads relevant journal data from DynamoDB, calls Groq API, and saves AI results back to DynamoDB.

Monitoring and Cost Guardrails

  • CloudWatch Logs, Metrics, and Alarms monitor Lambda errors, duration, throttles, and SQS backlog.
  • SNS sends alert notifications.
  • AWS Budgets alerts when spending reaches configured thresholds.
  • API Gateway throttling and Lambda reserved concurrency limit accidental high usage.

5. Technical Implementation

Main Features

  • Cognito registration, login, logout, and Google sign-in.
  • Journal CRUD with structured learning fields.
  • S3 image upload using presigned URLs.
  • AI report generation through SQS and Lambda Worker.
  • Quiz generation and quiz attempt scoring.
  • Search endpoint for journal discovery.
  • Analytics endpoint for progress dashboard.
  • Amplify deployment with environment variables.
  • WAF, throttling, CloudWatch alarms, SNS, and Budgets.

Core AWS Services

  • Amplify Hosting: frontend deployment.
  • CloudFront + WAF: public delivery and edge protection.
  • Cognito: authentication and JWT tokens.
  • API Gateway: HTTP API entry point.
  • Lambda: API handler, AI worker, search handler, analytics handler.
  • DynamoDB: serverless NoSQL data store.
  • S3: image storage.
  • SQS + DLQ: asynchronous AI job processing.
  • CloudWatch + SNS + Budgets: monitoring and cost control.
  • Groq API: AI completion for reports and quizzes.

7. Budget Estimation

For a small internship/demo environment, the expected cost is low because most services are serverless and pay-per-use.

  • Lambda, API Gateway, SQS, DynamoDB, S3, CloudWatch, SNS, and Amplify can remain within a few USD per month for light traffic.
  • Groq API usage depends on the number of AI reports and quizzes generated.
  • The recommended demo budget is USD 5-10/month, with AWS Budgets alerts at 50%, 80%, and 100%.

Cost controls include:

  • API Gateway throttling on AI routes.
  • Reserved concurrency for the AI Worker.
  • SQS DLQ to avoid infinite retry loops.
  • S3 lifecycle rules for uploaded images.
  • CloudWatch alarms for error rate and queue backlog.

8. Risk Assessment

Cost Overrun

AI generation and repeated API calls can increase cost. Mitigation: throttling, concurrency limits, Budgets alerts, and clear frontend retry behavior.

Sensitive Learning Data

Journal logs may contain personal notes or screenshots. Mitigation: Cognito authentication, JWT authorizer, private S3 bucket, presigned URLs, and least-privilege IAM.

AI Output Quality

AI reports or quizzes may be incomplete or inaccurate. Mitigation: structured prompts, JSON-only output, validation in Lambda Worker, and user feedback.

Operational Complexity

Multiple serverless components can be hard to troubleshoot. Mitigation: CloudWatch logs, DLQ inspection, clear naming conventions, and documentation.

9. Expected Outcomes

  • A working AI-assisted learning journal application.
  • Secure authentication and per-user data isolation.
  • Fast frontend experience through Amplify and CloudFront.
  • Scalable backend using API Gateway, Lambda, DynamoDB, S3, and SQS.
  • AI reports and quizzes generated asynchronously through Groq.
  • Monitoring and cost guardrails suitable for an internship/demo environment.
