1. First Cloud AI Journey - Level Up Your AWS Skills with Cloud Quest and Floci

A community-driven, hands-on technical meetup focused on helping beginners - especially students and junior developers - start their AWS journey without the fear of large bills. The session walked through two practical tools (AWS Cloud Quest and open-source Floci) and showed how combining them creates a complete, low-cost learning roadmap for real cloud skills.

Speakers

  • Huynh Thai Linh – AWS community learner and presenter, sharing a personal AWS

Summary of content and main activities

Huynh Thai Linh opened the session with a very relatable pain point: most beginners who try to learn AWS hands-on end up stressed about checking the billing dashboard every hour, forgetting to delete resources and accumulating surprise charges. The talk then reframed this problem by introducing a complete, zero-cost (or near-zero-cost) learning path built on two complementary tools: AWS Cloud Quest for guided, game-based architecture practice, and open-source Floci for lightning-fast local AWS emulation. The session finished with a clear 3-phase roadmap and an honest discussion of where each tool falls short.

  • Activity 1 — What Beginners Worry About in AWS

    • Framed the core problem: students constantly check the AWS bill, worry about forgetting to delete resources (EC2, RDS, NAT Gateway, Elastic IPs), and fear extra charges from idle services. Established why beginners avoid hands-on labs - and why that blocks real learning.

  • Activity 2 — The Three AWS Learning Tool Categories

    • Mapped the three ways to practice AWS today: (a) real AWS account - high realism but high cost; (b) commercial emulators like LocalStack Pro - paid, sometimes slow; (c) free open-source emulators - cost-free but with feature limits. Set up the trade-off space.

  • Activity 3 — AWS Cloud Quest: A Free Hands-on Learning Playground

    • Introduced AWS Cloud Quest as a free, game-based, guided learning environment. Highlights: no credit card billing stress, structured quests for beginners, focused on building the AWS architecture mindset rather than memorising service lists.

  • Activity 4 — Why Cloud Quest Works for Beginners

    • Three reasons: (1) guided AWS practice - no need to design labs from scratch; (2) easy entry for beginners with zero AWS background; (3) learning through game quests keeps motivation high. Positioned Cloud Quest as the ‘mindset & architecture’ phase.

  • Activity 5 — Open-Source Floci: Local AWS Testing

    • Introduced Floci as an open-source AWS service emulator that runs entirely on the local machine. Key benefits: AWS service emulation, zero cloud costs, fast architecture testing during development, ideal for coding and CI pipelines.

  • Activity 6 - Floci vs. LocalStack: Performance & Coverage

    • Presented the head-to-head benchmark. Floci is up to 138x faster than LocalStack on common operations, uses up to 11x less memory and storage, and supports more free services than the LocalStack Community edition - making it a better default for student projects.

  • Activity 7 - Floci in the Real Workflow

    • Showed how Floci fits into a developer loop: write Infrastructure-as-Code or app code locally, spin up emulated AWS resources on the laptop, test interactions in seconds, then push to a shared sandbox only when the design is stable. Highlighted cost and time savings.

  • Activity 8 - Effective Practical Learning Roadmap (3 Phases)

    • Combined the two tools into a single roadmap. Phase 1 - Mind & Architecture with Cloud Quest; Phase 2 - Code & Fast Testing with Floci; Phase 3 - Real Deployment & Production on real AWS. Students train the right skills with the right tool at the right cost level.

  • Activity 9 - Honest Discussion of Floci Limitations

    • Listed three real limitations: only a limited set of AWS services is supported, some services return fake or mock results instead of identical behaviour, and it cannot fully replace testing on real AWS for production workloads. Set correct expectations for students.

  • Activity 10 - Cost-Saving Mindset for Student AWS Labs

    • Wrapping principle: a student should avoid running 24/7 services on real AWS, prefer Cloud Quest for guided learning, use Floci for local iteration, and only deploy to real AWS for final demos or graded milestones. The roadmap makes AWS learning financially safe.

  • Activity 11 - Q&A and Community Discussion

    • Open floor where students asked about emulator accuracy, pricing for real AWS sandboxes, how to migrate Floci designs to real AWS, and which Cloud Quest roles to start with for a Solutions Architect career path.

Results and values ​​achieved

  • Lessons learned

    • Cloud hands-on without fear is possible - choosing the right tool for each learning phase is the real unlock, not avoiding AWS altogether.

    • Cloud Quest is best for building the architecture mindset: it is guided, gamified and zero-cost - perfect to start before touching Infrastructure-as-Code.

    • Floci is best for fast iteration: 138x faster than LocalStack and 11x more memory-efficient makes it a serious productivity booster for student projects and CI pipelines.

    • Cost anxiety is a design problem - using Cloud Quest + Floci + scripted teardown of real AWS labs is the practical answer for students on a tight budget.

    • Emulators are not magic: limited services, mock results and imperfect fidelity mean real AWS will still be the final test - but you only go there once the design is solid.

    • Tool layering > tool switching: a 3-phase roadmap (mindset, code, real AWS) is far more effective than picking one emulator or one sandbox and forcing every learning goal into it.

  • Newly Acquired Skills

    • Cost-Aware AWS Planning: ability to design hands-on labs in tiers - free tier / emulator / paid sandbox - and pick the right tier for each task. Useful for every future student project and group capstone.

    • AWS Architecture Mindset: mental model for thinking in services, regions, quotas and dependencies, built through Cloud Quest quests rather than dry document reading.

    • Local AWS Emulation: hands-on familiarity with Floci - how to spin up emulated AWS resources on a laptop, run interactions locally, and iterate without spending credits.

    • Tool Benchmarking: ability to compare AWS testing tools on real dimensions (speed, memory, supported services, fidelity) - a skill that also applies to many other engineering tool choices.

    • Practical Roadmap Design: the meta-skill of breaking a learning goal into phases and matching the cheapest, most effective tool to each phase - directly applicable to self-study and to leading small study groups.

    • Soft Skills: sharper active listening during English technical talks, structured note-taking on diagrams and benchmark tables, and asking clearer questions about pricing, accuracy and migration between tools.

Closing Note

Huynh Thai Linh’s talk was a refreshing reminder that learning AWS as a student is not about who can afford the biggest sandbox account - it is about choosing the right tool for the right phase of learning. By combining AWS Cloud Quest for the architecture mindset and open-source Floci for fast local testing, students can train almost every real-world AWS skill for free, and only graduate to a paid AWS environment when the design is truly production-ready. As a student I walked away with a clearer, cheaper learning roadmap, more confidence in self-directed cloud study, and a concrete playbook I can share with my team and the wider student AWS community.

2. The Iceberg of Procrastination - When Laziness is Just the Tip of Fear

A personal, science-backed soft-skill talk delivered inside an AWS Community Day event in Ho Chi Minh City. The session reframed procrastination - not as a time-management problem - but as an emotional regulation problem caused by three hidden layers of fear. The presenter, Khac Uy Pham, then layered on a simple, practical framework (the 5-Minute Rule) and tied everything back to the AWS builder mindset.

Speakers

  • Khac Uy Pham – 3rd Year student at Vietnamese-German University.

Summary of content and main activities

Khac Uy opened with three painfully honest questions: Has anyone NEVER procrastinated on an important moment? Has anyone scrolled the night before a deadline? Has anyone watched an opportunity - even a personal one - pass by without acting? He then drew the iceberg: only 20-30% of procrastination is the ’lazy’ surface everyone sees (scrolling TikTok, turning off alarms, saying ‘I’ll do it tomorrow’); the hidden 70-80% is fear - of not being good enough, of being judged and of failing. The talk then built a small but powerful toolkit: name the fear, start with five minutes, and let momentum carry the rest, all tied back to the AWS builder philosophy of ‘go build, fail fast, deploy fast, fix early’.

  • Activity 1 — Opening: A Familiar Enemy

    • Three interactive questions to break the ice: anyone who never procrastinated, anyone who scrolled before a deadline, anyone who stayed silent when they should have spoken. Established that procrastination is a shared human problem, not a personal failing.

  • Activity 2 — The Iceberg Visual Metaphor

    • Introduced the iceberg model that anchors the whole talk. The visible tip (laziness) is only 20-30% of the real cause; the hidden mass underwater (fear) is 70-80%. This reframe is the core thesis: ’laziness’ is a surface symptom, not the root cause.

  • Activity 3 — Surface Symptoms of Procrastination

    • Catalogued the ‘visible’ lazy behaviors: scrolling TikTok / Facebook, turning off alarms and snoozing, saying ‘I’ll do it tomorrow’, low-energy study sessions, last-minute panic. The point: these behaviors are what people see and judge themselves for, but they are not the real driver.

  • Activity 4 — The Hidden Fear: Emotional Regulation, Not Time Management

    • Cited Prof. Fuschia Sirois (University of Sheffield) and the APA: 70-80% of procrastination is an emotional regulation problem, not a time management problem. Over 80% of adults procrastinate in critical areas like health and career. Biologically, it is the Prefrontal Cortex (logic) vs the Limbic System (emotion), and emotion usually wins.

  • Activity 5 — Layer 1 of Fear: Fear of Not Being Good Enough

    • Explained the ‘inadequacy’ fear. When a student opens a complex roadmap (Kubernetes, Terraform, a capstone project) and feels overwhelmed, the brain closes the tab and switches to TikTok - instant relief, zero risk of feeling ’not smart enough’. The task never gets started because starting means confronting the feeling.

  • Activity 6 - Layer 2 of Fear: Fear of Judgment

    • Explained the social-evaluation fear: ‘You’re a Year-3 student and write code like this?’ Many students would rather not submit, not push code, not ask a question in class than risk negative judgment from peers, lecturers or recruiters. Silent procrastination.

  • Activity 7 - Layer 3 of Fear: Fear of Failure

    • Explained the ‘verdict’ fear: trying and failing feels like a permanent definition of who you are, so not trying is safer. Ironically, not trying is also a verdict - the only difference is that it happens quietly. The feared verdict is delivered either way.

  • Activity 8 - The Guilt Loop

    • Traced the self-reinforcing cycle: avoid the hard task to feel temporary relief, the relief is followed by guilt, the guilt becomes self-blame, the self-blame becomes stress, and the stress makes the next start even harder - fueling more avoidance. Made the cycle visible so people can step out of it.

  • Activity 9 - The Solution: Action Comes First, Confidence Comes Second

    • Cited Prof. Timothy Pychyl: emotion follows action, not the other way round. Start before you feel ready; confidence is a by-product of starting, not a prerequisite. This single insight inverts most students’ approach to hard tasks.

  • Activity 10 - Step 1: Name Your Fear

    • Practical communication refactor. Instead of saying ‘I am lazy’, say ‘I am afraid my code will be judged’, or ‘I am afraid of getting a low grade’, or ‘I am afraid of looking stupid in the demo’. Naming the fear separates ‘I am the problem’ from ‘a specific fear is the problem’ - and a specific fear is something you can act on.

  • Activity 11 - Step 2: The 5-Minute Rule

    • The most concrete tool from the talk. Pick one tiny starter action (open the laptop, write three lines of code, read one AWS doc page) and commit to only five minutes. Around 90% of the time, once the starter action is done, momentum takes over automatically and you end up working far longer than five minutes. Removes the ‘I have to finish it’ pressure that causes the original avoidance.

  • Activity 12 - The Speaker’s Own ‘YES’ Moment

    • Khac Uy shared the personal story behind the talk: he was invited to speak at an AWS event, felt completely inadequate, almost said no, paused, recognized that the ‘I am not good enough’ feeling was just the iceberg talking - and said yes anyway. He reframed his own fear, used the 5-Minute Rule to start writing the slides, and shipped the talk you just watched.

  • Activity 13 - How This Connects to the AWS Builder Mindset

    • Mapped the talk to AWS culture. ‘Go Build’ = start even if it is small. ‘Fail fast’ = lower the cost of being wrong. ‘Deploy fast’ = ship a rough version, don’t wait for perfect. ‘Fix early’ = iterate after deploying, not before. The framework professional builders use is the same framework students can use to defeat procrastination.

Results and values ​​achieved

  • Lessons learned

    • Procrastination is an emotional problem wearing a ’time management’ disguise - beating it starts with beating the emotion, not the calendar.

    • The ’laziness’ story we tell ourselves is almost always wrong - the iceberg is fear, and fear is something you can name, examine and shrink.

    • Confidence does not come before action - it comes after. Waiting to feel ready is the trap; the only way out is to start small and let the feeling catch up.

    • A 5-minute commitment is enough to break the guilt loop. The barrier to starting was never the size of the task - it was the size of the commitment in our heads.

    • Failing has to be reframed as a verb (‘I failed this demo’), not an identity (‘I am a failure’). Identity-based fear is what makes students avoid tasks for years.

    • Community events like AWS Community Day are a force-multiplier - by simply showing up, you have already broken the start barrier, and you can chain small actions on top of it.

  • Newly Acquired Skills

    • Fear-Name Reframing: the ability to translate vague ‘I am lazy’ self-talk into a specific, nameable fear (of judgment, of failure, of not being good enough), which separates identity from behaviour and makes the fear addressable.

    • 5-Minute Rule Habit: a concrete, repeatable micro-commitment framework that can be applied to coding assignments, AWS lab prep, fitness routines, group reports and any task we are avoiding.

    • Emotional Awareness: better recognition of the guilt-loop signals - instant relief, blame, stress, more avoidance - and the ability to step out of the cycle at the first step instead of the fourth.

    • Self-Compassion in Tech: learned to treat AWS ‘fail fast’ as an instruction to be kind to my own beginner attempts, rather than as pressure to perform. Lowered the emotional cost of being wrong.

    • Public-Speaking Courage: internalised the speaker’s own ‘YES’ story and now keep an eye out for similar small opportunities (lightning talks, class demos, community posts) that would normally be skipped.

    • Soft Skills: sharper listening during a personal, story-driven talk; structured note-taking on a framework that blends psychology, biology and engineering culture; and asking clearer questions about how to apply the framework to my own pending tasks.

Closing Note

Khac Uy Pham’s talk hit a nerve that technical talks rarely do. It told students the truth we already half-know: the reason we are not shipping our side projects, not finishing our AWS certifications and not speaking up in class is not laziness - it is an iceberg of fear that we keep calling something else. By giving us a vocabulary (the three fears), a diagnostic (name your fear) and a starter action (the 5-Minute Rule), the session turned an emotional problem into an engineering problem - the kind I can actually solve. As a student I walked away with a calmer story about who I am when I procrastinate, a habit I can apply the same day, and a community reminder that ‘go build’ starts with the next five minutes, nothing more.

3. Why We Always Need Confidence - A Must-Have for Our Careers, Campus Life, and Everything in Between

A student-centered soft-skill talk inside the First Cloud AI Journey community track, focused on reframing confidence as a learnable skill rather than an innate trait. The session walked through the cost of self-doubt in student life, the psychology behind imposter syndrome and the Dunning-Kruger effect, the practical benefits of confidence for communication, collaboration and ownership, and a concrete everyday toolkit (preparation, small wins, Mel Robbins’ 5-Second Rule) for building self-trust one moment at a time.

Speakers

  • Nguyen Thi Quynh Nhu – student presenter and community learner.

Summary of content and main activities

Nguyen Thi Quynh Nhu opened with a Mark Twain quote about courage being resistance to fear, not absence of fear, then anchored the talk in a deeply relatable student pain point: ‘I knew the tech, but I could not present it’ - strong technical understanding, nervous delivery, underestimated results. She then redefined confidence as ’trying even when nervous’, explicitly not ego and not perfection, framed confidence as a buildable skill, and walked the audience through the science (Imposter Syndrome, Dunning-Kruger, Carol Dweck’s growth mindset) before closing with a small, repeatable daily toolkit: prepare, celebrate small wins and use Mel Robbins’ 5-Second Rule.

  • Activity 1 — Setting the Frame with Quotes

    • Opened with Mark Twain: ‘Courage is resistance to fear, mastery of fear - not absence of fear.’ Paired it with Amelia Earhart’s ‘The most difficult thing is the decision to act - the rest is merely tenacity.’ Established that confidence is acting, not feeling ready, before defining any framework.

  • Activity 2 — The Familiar Student Pain: ‘I Knew The Tech, But I Couldn’t Present It’

    • Shared a candid personal story: solid technical understanding, nervous presentation, underestimated results. Made it explicit that this is a near-universal student experience in labs, demos, capstone reviews, scholarships and interviews, and is the reason the talk exists at all.

  • Activity 3 — Redefining Confidence (Not Ego, Not Perfection)

    • Three deliberate reframes. Confidence is NOT ego - it is not about being the loudest person in the room. It is NOT perfection - you do not need every answer before you speak. It IS trying even when you feel nervous. This reframe separates posturing from quietly doing the thing anyway.

  • Activity 4 — Technical Skills Are the Foundation, Confidence Is the Bridge

    • Used a metaphor from the rest of the First Cloud AI Journey: students spend hours on labs, Terraform, Bedrock and dashboards - those technical skills are the foundation, but confidence is the bridge that actually carries those skills out into jobs, internships, group projects and scholarship interviews. Without the bridge, the foundation stays invisible.

  • Activity 5 — he Cost of Self-Doubt in Student Life

    • Mapped three ‘hidden’ costs students usually do not name. (1) Missed chances: opportunities never even attempted - scholarships, hackathons, talks, internships. (2) Invisible pressure: internal stress that looks calm from the outside but is exhausting. (3) Hidden potential: skills the student has but never shows because they did not raise their hand.

  • Activity 6 — The Science: Imposter Syndrome

    • Introduced Imposter Syndrome as the persistent feeling of ‘I don’t deserve this’ - even after being admitted to a programme, landing an internship or passing an interview. Showed how it makes students shrink, apologize for their work and avoid new opportunities, magnifying the hidden costs listed in Activity 5.

  • Activity 7 - The Science: Dunning-Kruger Effect

    • Introduced the Dunning-Kruger Effect as a cognitive bias where beginners feel overconfident and intermediate learners suddenly feel ‘stupid’ as they see how much more there is to learn. Reassured the audience: this dip is a normal stage of real learning, not a sign of being incapable. It is the cost of moving from surface knowledge to deep knowledge.

  • Activity 8 - Carol Dweck’s Growth Mindset

    • Discussed caching for availability, built-in failover to secondary origins, graceful failure with custom error pages, multi-layer caching (PoP -> Regional Edge Cache -> Origin Shield), HTTP/3 (QUIC/UDP) multiplexing and AWS global backbone for persistent connections.

  • Activity 9 - The Power of Confidence: Connections (Communicate, Collaborate, Connect)

    • First pillar of confidence’s practical value. Confidence enables you to communicate ideas clearly (presentations, stand-ups, code reviews), collaborate on equal footing with peers and seniors, and connect with mentors, judges and recruiters. Without it, even good work stays isolated.

  • Activity 10 - The Power of Confidence: Ownership

    • Second pillar. Confidence lets you make decisions in ambiguous situations, defend your ideas in a respectful way (even when challenged), and learn from mistakes without collapsing into self-blame. This is what turns a contributor into an owner in a team.

  • Activity 11 - The Power of Confidence: People Buy the ‘Why’ (Simon Sinek)

    • Third pillar. Quoted Simon Sinek: ‘People don’t buy what you do, they buy why you do it.’ Why you do something only lands if it is delivered with confident clarity. Confident delivery is therefore a multiplier on every resume line, side project and AWS certification you earn.

  • Activity 12 - Hack #1: Preparation as a Confidence Builder

    • First practical tool. Preparation creates clarity for presentations, interviews and projects, which reduces fear of the unknown. Concrete examples: rehearse out loud, write your opening line first, walk through the demo before the day of the talk.

  • Activity 13 - Hack #2: Celebrate Small Wins

    • Second practical tool. Build self-trust by acknowledging small acts: asking a question in class, fixing a bug, sharing an idea in a team meeting, finishing a small AWS lab. Small wins are the raw material confidence is built from, not big milestones.

  • Activity 14 - Hack #3: The 5-Second Rule (Mel Robbins)

    • Third and most actionable tool. When an opportunity appears (raise your hand, send the application, submit the demo) and self-doubt shows up, count backwards 5-4-3-2-1 and move physically. The countdown interrupts the hesitation loop and turns intention into action. ‘The rest is merely tenacity.’

  • Activity 15 - Wrap-Up and Final Message

    • Closing synthesis. Confidence is not something you are born with - it is something you build, one prepared moment and one small win at a time. The talk closed with Brené Brown’s reminder that courage and confidence are verbs - they happen in the small act of moving, not in feeling ready. Final call to action: ‘Act.’

Results and values ​​achieved

  • Lessons learned

    • Confidence and courage are not the absence of fear - they are the decision to move while fear is still there. Most students wait for the fear to disappear; it almost never does on its own.

    • Self-doubt has costs I was not naming - missed chances, invisible pressure and hidden potential - and naming them is the first step to stopping them.

    • The ‘I don’t deserve this’ voice (Imposter Syndrome) is a normal cognitive pattern, not evidence that you actually don’t deserve it. Treating it as data, not verdict, is a real shift.

    • The mid-learning dip (Dunning-Kruger) means I am actually getting deeper, not getting worse. Re-framing that dip changes how I persist through hard AWS topics like networking, IAM or Terraform.

    • Carol Dweck’s growth mindset and confidence are the same posture - ‘I can get better’ is inherently more confident than ‘I have to prove I am already great’.

    • Confidence is a verb you can practise in five minutes (the 5-Second Rule), not a personality trait you hope to inherit.

  • Newly Acquired Skills

    • Confidence Reframing: the ability to translate ‘I am not confident’ into a specific barrier (nervous delivery, unprepared slot, fear of judgment) and pick the matching hack. Confidence stops being a abstract character trait and becomes a concrete fix.

    • Imposter Syndrome Awareness: recognising the ‘I don’t deserve this’ voice in myself and in teammates as a common pattern, and learning to talk back to it with statements like ’lots of qualified people feel this, it is not evidence’. Useful in interviews, scholarship reviews and group presentations.

    • Dunning-Kruger Navigation: practical understanding of the mid-learning dip and the discipline to keep pushing through it instead of concluding ‘I am bad at this’. Especially useful for AWS, DevOps and certification study where intermediate material feels overwhelming.

    • Growth Mindset Communication: learned to talk about skills in ‘I can get better’ language rather than ‘I am a person who is/can’t’ language, both in self-talk and when giving feedback to teammates.

    • 5-Second Rule Practise: a concrete, body-based intervention (count 5-4-3-2-1, then move) that interrupts hesitation in real time - already used to send in a class application and to raise a hand in Q&A.

    • Soft Skills: better active listening during a personal, story-driven talk; structured note-taking on psychology frameworks; asking clearer questions about how to support peers who show low confidence in group settings.

Closing Note

Nguyen Thi Quynh Nhu’s talk was a quiet but firm reminder that confidence is not something I either have or do not have - it is something I have been quietly building or quietly eroding with every small act of showing up or staying silent. By naming Imposter Syndrome and the Dunning-Kruger dip, by quoting Carol Dweck, Simon Sinek and Brené Brown, and by handing us the 5-Second Rule as a tiny but real tool, the session turned an emotional topic into one I can practise - in the next class, the next stand-up and the next scholarship application. As a student I walked away with a kinder story about who I am when I am nervous, a habit I can apply the same day, and a clearer responsibility to support classmates who are visibly carrying the same invisible weight.

4. The Hidden Iceberg of a Project - DevOps Before Disaster

A technical-and-culture talk that uses the iceberg metaphor to explain why software projects fail in slow, painful ways - and how DevOps, understood as a mindset (People, Process, Technology) and not just a toolset, can address the hidden causes below the surface before they turn into a project-level disaster. The session blends systems thinking, DevOps principles and very concrete tooling (Docker, Kubernetes, CI/CD, Cloud, Terraform) with candid advice on team culture and ownership.

Speakers

  • Tran Minh Quan – community speaker and DevOps practitioner.

Summary of content and main activities

Tran Minh Quan opened with a painfully familiar story: a feature that was estimated at two days triggered three reworks, endless misunderstandings and team-wide exhaustion. He used that story to introduce the iceberg model of a project - in which only symptoms (missed deadlines, bugs, failed deployments, complaints, burnout) are visible above the water, while the true causes (ambiguous requirements, communication gaps, siloed teams, lack of ownership, manual processes, slow feedback loops) stay underwater and slowly sink the project. He then contrasted the symptom-fixing mindset with system thinking, framed DevOps as People + Process + Technology (not just tooling), laid out four pillars (collaboration, automation, fast feedback, continuous improvement), mapped each hidden cause to its DevOps solution and closed with the Titanic warning: small ignored warnings become big disasters if you only fix what’s visible.

  • Activity 1 — Opening Story: The ‘Simple Feature’ That Ate the Sprint

    • Set the tone with a real scenario: two-day estimate, three reworks, endless misunderstandings and a project that feels exhausted. Made the audience recognise their own team in the story before the iceberg model was introduced.

  • Activity 2 — Framing the Iceberg Model of a Project

    • Introduced the central analogy. Above water: the symptoms the manager and the customer see. Below water: the slow, structural causes that quietly accumulate. Without diving below the surface, no amount of firefighting can save the project.

  • Activity 3 — The Tip of the Iceberg - Visible Symptoms

    • Catalogued what teams usually blame: missed deadlines, production bugs, failed deployments, customer complaints, team burnout. These are real - and painful - but they are the output of the system, not the cause.

  • Activity 4 - Beneath the Surface - Hidden Causes

    • Mapped six structural causes: requirement ambiguity (vague tickets), communication gaps (handoff losses between dev, QA and ops), siloed teams (dev vs. ops vs. QA), lack of ownership (no one owns the whole flow), manual processes (deploys by hand), slow feedback loops (bugs caught weeks after commit).

  • Activity 5 — Why Solving Symptoms Is Not Enough

    • Contrasted two mindsets. Firefighting / Symptom Fixing = ‘fix the deadline, fix the bug, fix the deploy’. System Thinking = ‘why did the deadline slip? why did the bug reach production? why did the deploy break?’. Without the second, the same symptoms return in a different shape every sprint.

  • Activity 6 — DevOps as the Solution: People, Process, Technology

    • Defined DevOps not as a tool list but as a triangle. People: culture, ownership, shared responsibility. Process: how work moves from idea to production. Technology: the tools that support the people and process. Quoted the speaker’s reminder that you can automate deployments but you cannot automate trust, communication or ownership.

  • Activity 7 - Pillar 1: Collaboration (Breaking Silos)

    • First pillar. Move from blame culture (dev blames QA, QA blames ops, ops blames dev) to shared responsibility and shared goals. Practical signals: cross-functional teams, joint incident reviews (not post-mortems designed to assign blame), shared Slack channels for product + infra.

  • Activity 8 - Pillar 2: Automation (Reducing Friction and Human Error)

    • Second pillar. Reduce manual hand-offs through CI/CD, Infrastructure as Code, automated tests and automated workflows. The point is not ‘more scripts’ but ‘fewer places a human can forget a step’ - reworks disappear when the pipeline enforces quality.

  • Activity 9 - Pillar 3: Fast Feedback (See Problems Earlier)

    • Third pillar. Push discovery as early as possible through monitoring, observability, continuous integration, feature flags and canary releases. The cost of finding a bug grows exponentially with time - fast feedback collapses that cost back down.

  • Activity 10 - Pillar 4: Continuous Improvement (Learning, Not Repeating)

    • Fourth pillar. Treat failures as data. Use retrospectives, blameless post-mortems and concrete metrics to learn from incidents and near-misses. The opposite culture - hiding mistakes and re-running the same playbook - is what sinks projects slowly.

  • Activity 11 - Mapping Hidden Causes to DevOps Solutions

    • Drew an explicit cause-to-pillar map: requirement ambiguity is reduced by collaboration (shared refinement); communication gaps by collaboration (joint channels, demos); siloed teams by shared ownership; manual processes by automation; slow feedback loops by fast feedback; repeated failures by continuous improvement. Made the framework operational, not just philosophical.

  • Activity 12 - Tooling Reality Check (Docker, Kubernetes, CI/CD, Cloud, Terraform)

7. How Cloud Architecture Powers Modern Multiplayer Games

A technical deep-dive seminar inside the First Cloud AI Journey community track, delivered by Pham Quang Thai. The talk framed the explosive growth of online multiplayer gaming and the engineering problem it creates: simultaneously demanding real-time interaction with minimal latency AND the ability to instantly scale to millions of concurrent players. The seminar dissected infrastructure models from Amazon Web Services (GameLift, FlexMatch, FleetIQ), Google Cloud Platform (GKE, Agones with Ubisoft, Quilkin), Microsoft Azure (complex demand forecasting) and Oracle Cloud, and closed with future trends in edge computing, cloud gaming with GPU servers and Generative AI for virtual worlds.

Speakers

  • Pham Quang Thai

Summary of content and main activities

Pham Quang Thai opened with a clear engineering picture of the modern multiplayer industry and the dual pressure it puts on infrastructure: real-time interaction with minimal latency AND elastic scaling for millions of concurrent sessions. He categorised the problem into two architectural shapes - session-based (FPS, MOBA, Battle Royale, low-latency dedicated game servers) and persistent-world (MMOs, stateful, AI-heavy) - and walked through the AWS (GameLift, FlexMatch, FleetIQ), Google Cloud (GKE, Agones, Quilkin), Microsoft Azure and Oracle Cloud stacks that power them today. He spent the middle of the talk on the matchmaking flow (edge latency via ping endpoints stored in DynamoDB, API Gateway plus JWT authentication, AWS Lambda decentralised processing, RTT minimisation), the elasticity pattern via Agones ‘Allocated’ states and Spot / Preemptible instances managed by a FleetIQ-style predictive algorithm, and the security stack (REPs at the edge, DynamoDB rate-limiting writes at up to 800,000 writes per 30 seconds, Kinesis to S3 plus Athena SQL queries for cheater detection). He closed with real-game references (Valorant and the Tran Huu Nghia ‘SouhcNi’ mention on millisecond latency jitter) and three forward-looking vectors: edge computing (CloudFront), cloud gaming with GPU-equipped servers under 100 ms interaction latency over RTP/SRTP, and Generative AI for virtual-world creation powered by Amazon SageMaker.

  • Activity 1 — Setting the Industry Stage: The Multiplayer Pressure Problem

    • Opened with the modern multiplayer industry’s growth wave and the dual engineering constraint it creates: real-time interaction with minimal latency AND instant scaling to handle millions of concurrent players. The audience appreciated that a single bad architecture choice is now visible to every player, every minute.

  • Activity 2 — Categorising Game Architecture - Session-Based

    • First architectural shape. Session-Based Architecture focuses on rapidly creating Dedicated Game Servers (DGS), matchmaking based on user latency, transient state and battle-style flow. Typical genres: FPS, MOBA, Battle Royale. Strict latency target: 50 ms or less per round trip.

  • Activity 3 — Categorising Game Architecture - Persistent-World (MMO)

    • Second architectural shape. Sustainable World Architecture demands enormous CPU for AI-driven simulation of thousands of entities in a continuous world. MMORPG servers typically run physics and character-state simulation at a 30 Hz Tick Rate. Requires stateful databases for permanent storage of player assets, levels and inventory.

  • Activity 4 - AWS Stack Walkthrough - GameLift and FlexMatch

    • Dove into the AWS offering. Amazon GameLift manages the full server lifecycle - spin-up, replace, retire. FlexMatch runs custom matchmaking rules (latency, skill, team composition). The combination is the de-facto backbone of many Western session-based games.

  • Activity 5 — Matchmaking Flow Step-by-Step (Latency and Edge Detection)

    • Walked through the matchmaking pipeline. Step 1: edge latency measurement - the client probes endpoints stored in DynamoDB. Step 2: SDK request initiation via API Gateway with JWT authentication as a JSON Web Token. Step 3: AWS Lambda runs decentralised matchmaking logic. Step 4: optimal region chosen via a mathematical RTT (Round-Trip Time) minimisation formula selecting the cloud region with the lowest average latency across the group.

  • Activity 6 - Elasticity Pattern: Agones ‘Allocated’ State on Kubernetes

    • Explained the killer Agones pattern. When a session is matched, Agones flips the GameServer into an ‘Allocated’ state, which prevents Kubernetes from scaling it down during an active match. Quilkin, an open-source UDP proxy, supports low-latency real-time traffic. Together they solve the race condition between scale-down and active match.

  • Activity 7 - Cost Optimisation: FleetIQ, Spot and Preemptible Instances

    • Showed the cost lever. FleetIQ continuously runs predictive algorithms to assess the stability of Spot partitions, then places game servers on the cheapest reliable Spot capacity. Reported saving: 70 to 80 percent vs on-demand. Critical insight: saving money on game infrastructure is a continuous stability-vs-cost tradeoff, not a one-time deal.

  • Activity 8 - Google Cloud, Azure and Oracle Cloud Differentiators

    • Brief on the other major clouds. Google Kubernetes Engine plus Agones allows multi-cloud, fully open-source, vendor-locked-free deployments. Azure focuses on complex demand forecasting models to optimise inventory and resource allocation, borrowed from large-scale supply chain systems. Oracle Cloud is a competent alternative for cost-sensitive deployments with strong database roots. Cross-cloud flexibility is a real strategic lever for studios.

  • Activity 9 - Engineering Latency Budget (50 ms for Session-Based)

    • Distilled the latency target. For session-based games, the practical latency budget is 50 ms or less per round trip. Anything beyond that consistently produces a poor player experience, even with great graphics. The closest reference to this is the SouhcNi (Tran Huu Nghia) mention from a Valorant case study about millisecond-level latency jitter.

  • Activity 10 - Security Layer 1: Remote Entry Points (REPs) as DDoS Shields

    • First security layer. Remote Entry Points (REPs) are deployed as boundary shields at the network edge. They hide the private VPC grid from the public internet, absorb DDoS attacks, and ensure that player traffic only reaches the real game servers after the REP has validated the session.

  • Activity 11 - Security Layer 2: DynamoDB Writes and Anti-Fraud

    • Second security layer. A distributed NoSQL database like Amazon DynamoDB records every state change - up to 800,000 writes per 30 seconds in a typical live game. Rate limiting at this layer prevents item duplication exploits (the ‘buy once, receive twice’ hack) by ensuring each transaction is uniquely recorded.

  • Activity 12 - Security Layer 3: Kinesis, S3 and Athena Cheat Detection

    • Third security layer. Standard data processing pipelines use Amazon Kinesis Data Streams to push telemetry into S3, then Amazon Athena runs SQL queries directly on that lake to analyse anomalous behaviour, detect hacks, cheats and transaction anomalies. Real-time observability plus retrospective forensics from the same data.

  • Activity 13 - Asset and State Synchronisation (Live Service Backbone)

    • Addressed the persistent problem. All changes to player assets, levels and equipment must be continuously synchronised between the client and the server across thousands of concurrent players. This is the operational definition of ’live service’ gaming and the source of most cheating, fraud and desync complaints.

  • Activity 14 - Real-World Anchor: The SouhcNi (Tran Huu Nghia) Reference

    • Pinned the latency discussion to a real esports reference. The SouhcNi case study on millisecond-level latency jitter in Valorant showed that for competitive players the difference between a good and a great match is literally a few milliseconds. Speakers, students and developers alike could relate to that intuition of ‘feel’ being driven by an engineering number.

  • Activity 15 - Future Trend 1: Edge Computing with Amazon CloudFront

    • First forward-looking slide. Move game logic closer to the player via edge nodes powered by Amazon CloudFront. Edge computing reduces the hop count and gives the lowest-possible latency for session-based games in the same way CloudFront already does for static content.

  • Activity 16 - Future Trend 2: Cloud Gaming with GPU Servers

    • Second future vector. Cloud gaming streams full GPU-rendered gameplay using RTP/SRTP protocols under a 100 ms interaction latency budget. Users on low-end devices can play AAA games because all the rendering happens on the cloud side. The infrastructure cost is high, but the addressable market (everyone with a phone) is enormous.

  • Activity 17 - Future Trend 3: Generative AI for Virtual World Creation

    • Third future vector. Generative AI is used to procedurally create game worlds, quests, characters and dialogue, often via Amazon SageMaker. The infrastructure implication: large bursts of GPU and CPU when an agent generates a new zone, then quiet loads again - a new kind of elastic pattern for cloud providers to support.

  • Activity 18 - Closing Synthesis: Resilience, Latency and Security Harmony

    • Closed with the integrator’s view. Cloud architecture for modern multiplayer games is a sophisticated ecosystem, not a single service. It requires harmony of resilience (Agones ‘Allocated’ state plus Spot stability via FleetIQ-style prediction), latency optimisation (edge computing, RTT minimisation, CloudFront) and rigorous security auditing (REPs, DynamoDB rate limiting, Kinesis + Athena pipelines).

Results and values ​​achieved

  • Lessons learned

    • A multiplayer game architecture is really two architectures - session-based (low latency, transient) and persistent-world (stateful, AI-heavy). Picking the wrong one is a multi-year mistake; picking correctly makes everything else easier.

    • Agones shows that a hard problem (Kubernetes scaling down a session mid-match) can be solved with one CRD state - just by introducing an ‘Allocated’ state that Kubernetes will not interrupt. Open-source solutions often beat closed.

    • Spot instances plus a predictive stability algorithm (FleetIQ-style) save 70 to 80 percent vs on-demand without sacrificing player experience - this is the single most useful cost tool for any student or indie game studio.

    • Matchmaking is a math problem as much as an engineering problem. The RTT minimisation formula, decentralised Lambda logic and edge-ping detection via DynamoDB together produce the difference between a player who waits 15 seconds and one who waits 3.

    • The 50 ms latency budget is not a guideline - it is the line. Above it consistently, the game feels broken. Below it, and combined with low jitter, the game feels magical. The esports reference to millisecond-level jitter makes this tangible.

    • Security for live-service games is best layered edge-to-inside: REPs at the boundary for DDoS, DynamoDB for rate-limited state writes, Kinesis + S3 + Athena for cheater detection. Each layer defends a different threat; skipping any one of them is what shows up in the news.

  • Newly Acquired Skills

    • Multiplayer Architecture Reading: ability to read a game company’s job description or tech blog and identify which of the two architectures (session-based vs persistent-world) it runs, which orchestration layer it uses and where its cost optimisation sits.

    • Game-Tech Cloud Stack Mental Model: a working map of AWS GameLift / FlexMatch / FleetIQ vs Google GKE + Agones + Quilkin vs Azure forecasting vs Oracle Cloud. Lets me pick the right stack on purpose instead of by accident.

    • Matchmaking Pipeline Mental Model: the ability to sketch a full matchmaking pipeline (edge-ping detection via DynamoDB, JWT-authenticated API Gateway, decentralised Lambda logic, RTT minimisation formula) for any future multiplayer side-project.

    • Spot + Predicted-Stability Pattern: practical knowledge of how to safely use Spot capacity with a predictive stability algorithm (FleetIQ-style) so cost savings of 70 to 80 percent do not cost us player trust.

    • Edge-to-Inside Security Pattern: a structured three-layer security shield (REPs at the edge for DDoS, DynamoDB for state-write rate-limiting, Kinesis + S3 + Athena for retrospective cheat detection) ready to copy into future live-service projects.
