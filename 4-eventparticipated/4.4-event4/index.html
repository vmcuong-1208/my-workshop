1. AgenticOps Platform - Autonomous Cloud Operations

A platform showcase session hosted inside the First Cloud AI Journey community track, delivered by CloudThinker JSC. The talk introduced the company’s positioning as the pioneer ‘VibeOps’ / AgenticOps platform - a system where specialised AI agents autonomously operate, secure and optimise AWS cloud infrastructure 24/7 - and walked through the four-module platform loop (Prevent, Resolve, Predict, Learn), the specialised agents (Cloud, Kubernetes, Security, Database, Super), the AWS integration (Bedrock, SaaS-on-AWS, BYOC), measurable outcomes (MTTR, auto-resolve rate, waste recovery, engineer toil), customer case studies across Telco, FSI, FinTech, Corporate and Manufacturing, and a transparent pricing/deployment model.

Speakers

  • CloudThinker JSC (cloudthinker.io). – Global Reach, Local Expertise - #1 AWS Agentic AI Consulting Service (new, 2026 narrative line).

Summary of content and main activities

CloudThinker opened with positioning and customer context (Telco, FSI, FinTech, Corporate, Manufacturing), then framed the problem: today’s cloud complexity is not an accident, it is the cumulative result of every right call the team made - and it has produced two visible groans, microservices complexity at scale and the ‘FSI paradox’ (more tools, more engineers, same MTTR). The talk then introduced the Enterprise AgenticOps Platform as a continuous loop of four modules - Prevent (AI Code Review), Resolve (Deep Response), Predict (Cloud Keepers), Learn (MemGraph) - backed by specialised AI agents (Cloud, Kubernetes, Security, Database, Super) running on AWS with Bedrock. The session closed with measurable outcomes (MTTR -87%, 70% auto-resolved, 15-25% cloud waste recovered, engineer toil 40% to <10%, zero-downtime EKS upgrades), case studies (F88, NextPay, Diaflow), a transparent pricing model (Team / Scale / Scale+ / Enterprise) and a deployment choice (CloudThinker Managed SaaS vs. Customer Managed / BYOC), all anchored in the dual pitch line ‘Global Reach - Local Expertise.’

  • Activity 1 — Brand Introduction: Autonomous Cloud Operations

    • Opened with the headline ‘Autonomous Cloud Operations.’ Established the company’s promise: specialised AI agents operate, secure and optimise AWS cloud infrastructure 24/7, with no human-in-the-loop for routine work. Set up the rest of the session.

  • Activity 2 — Positioning: Global Reach, Local Expertise

    • Defined CloudThinker’s market stance as a ‘Global Reach, Local Expertise’ company - international ambition but rooted in the specific realities of Vietnam’s enterprise and FSI customers.

  • Activity 3 — ‘VibeOps’ / AgenticOps Branding (New #1 AWS Agentic AI Service)

    • Introduced the coined term ‘VibeOps’ as the productisation of AgenticOps - a way of running cloud operations that feels (almost) autonomous. Stressed this is a ’new’ consulting positioning that they want to own in the AWS ecosystem.

  • Activity 4 — Customer Profile Slides (Telco, FSI, FinTech, Corp, Manufacturing)

    • Walked through the five customer verticals they serve - Telco, FSI, FinTech, Corp and Manufacturing. Showed real logos and short blurbs so the audience could map the platform to their own industry.

  • Activity 5 — Customer Challenge #1 - Microservices Complexity

    • Set up the first pain point. As customers modernise, they rightly adopt microservices, but the cumulative result is hundreds of services, dozens of data stores and a topology no human team can keep in their head at 3am. Highlighted this as the reason operations fatigue is now a C-level conversation.

  • Activity 6 - Customer Challenge #2 - The FSI Cloud Operations Paradox

    • Set up the second pain point, hardest in financial services: ‘More tools. More engineers. Same MTTR.’ Despite years of investment in observability and ITSM tools, Mean Time to Recovery has barely improved. Frame for the AgenticOps pitch below.

  • Activity 7 - ‘How We Got Here’ - Complexity as Cumulative, Not Accidental

    • Reframed the current state as the sum of good decisions over time (right choices to deploy more services, more tools, more people) - not as a failure of engineering. This empathetic framing prepared the room to listen instead of defend.

  • Activity 8 - Enterprise AgenticOps Platform Overview

    • Introduced the platform as a continuous loop of four core modules - Prevent, Resolve, Predict, Learn - supported by specialised AI agents running on AWS. Each module was then explained individually with concrete numbers.

  • Activity 9 - Module 1 - Prevent (AI Code Review)

    • First module. Scans pull requests for bugs and security gaps before merge. Outcomes shown: 5x faster reviews (minutes instead of days), 75% detection rate at the PR level, ~98% of the industry benchmark leaderboard. Positioned as a ‘shift left’ for the agentic era.

  • Activity 10 - Module 2 - Resolve (Deep Response)

    • Second module. Pinpoints root causes by correlating logs, metrics and traces across accounts and services. Reported numbers: 96% accuracy on root-cause identification, MTTD (Mean Time To Detect) under 5 minutes, MTTR under 10 minutes. This is the module that most directly tackles the FSI paradox.

  • Activity 11 - Module 3 - Predict (Cloud Keepers, Agentic FinOps)

    • Third module. Proactive FinOps agent that hunts for unused / underused / rightsizable resources. Pitched with a strong ‘skin in the game’ line: zero-fee if no savings are found (ROI-guaranteed). Direct answer to the cloud-waste anxiety common in FSI customers.

  • Activity 12 - Module 4 - Learn (MemGraph / Corporate Memory)

    • Fourth module. A graph-based memory system where every incident, every resolution and every reviewer comment is stored as context. Future agents pull from MemGraph to make better decisions - the platform ‘gets smarter the longer it runs.’

  • Activity 13 - The Specialised Agents (Cloud, Kubernetes, Security, Database, Super)

    • Detailed the agent roster. Cloud Agent for general AWS tasks. Kubernetes Agent for EKS and container platforms. Security Agent for posture, IAM and threat detection. Database Agent for RDS, Aurora and DynamoDB performance and schema work. Super Agent orchestrates the others - the ‘commander’ who decides which specialist to dispatch.

  • Activity 14 - AWS-Native Integration (Powered by Amazon Bedrock)

    • Showed that the platform is SaaS-on-AWS or BYOC (Bring Your Own Cloud), powered by Amazon Bedrock for the underlying LLM capabilities. Estimated token cost around $200 per active user. Made the platform’s ‘AWS-native’ positioning tangible, not marketing-flavored.

  • Activity 15 - Headline Outcomes (MTTR -87%, 70% Auto-Resolve, 15-25% Waste, Toil 40% to <10%)

    • Compiled the platform’s headline KPIs: MTTR reduced by 87%, 70% of incidents auto-resolved, 15-25% of cloud waste recovered, engineer toil cut from 40% to under 10%, and Zero-downtime production EKS upgrades. A compact, scannable summary slide before the case studies.

  • Activity 16 - Case Study 1 - F88 (FinTech, Multi-Account Cost Control)

    • F88 regained 100% control over multi-account AWS costs and reduced manual operations work by 80% within three months. Showcase of the Predict module’s ROI in a Vietnamese FinTech.

  • Activity 17 - Case Study 2 - NextPay (Payments, Zero-Downtime EKS Upgrade)

    • NextPay upgraded the entire EKS infrastructure in one month with zero downtime and optimised RDS costs along the way. Showcase of the Kubernetes and Database agents working together on a payments platform.

  • Activity 18 - Case Study 3 - Diaflow (Global AI, SOC2 / HIPAA / GDPR in 4 Weeks)

    • Diaflow automated multi-region infrastructure across three regions and met SOC 2, HIPAA and GDPR compliance in four weeks. Showcase of the Security Agent and the platform’s compliance-readiness for global AI products.

  • Activity 19 - Pricing Model (Team / Scale / Scale+ / Enterprise)

    • Transparent pricing block: Team tier at $25 per seat (200 credits), Scale tier at $100 per seat (1,000 credits), Scale+ tier at $200 per seat (2,000 credits) and Enterprise as a custom Private Offer. Credits consumption tied to the agent actions taken, not to seats alone.

  • Activity 20 - Deployment Choice (CloudThinker Managed vs. Customer Managed / BYOC)

    • Two deployment paths. CloudThinker Managed - the platform is a managed SaaS on AWS, fastest to start. Customer Managed (Enterprise / BYOC) - the platform runs inside the customer’s own AWS account, addressing data-residency, compliance and auditability concerns of regulated customers such as FSI.

  • Activity 21 - Close: Scale Together / Corporate Memory

    • Closing line. ‘Scale Together with CloudThinker.’ Final reminder that the platform’s MemGraph layer builds ‘Corporate Memory’ - the team, the agents and the institutional knowledge all working from the same evolving context. Call to action framed as a partnership, not a vendor relationship.

Results and values ​​achieved

  • Lessons learned

    • Agentic AI is now being productised as a real operations platform, not only shown in demos - and the four-module pattern (Prevent, Resolve, Predict, Learn) is a transferable mental model even for student projects.

    • Tooling and headcount do not automatically reduce MTTR (the FSI paradox) - what reduces MTTR is closing the loop between detection, decision and action, which is exactly what specialised agents do.

    • Cloud complexity is cumulative, not accidental - it is the sum of every right decision made over years. The right response is not ’less cloud’ but ‘smarter operators’, which is a much kinder and more accurate framing than ‘your architecture is bad’.

    • FinOps with an ‘ROI-guaranteed’ pricing model (no savings, no fee) is a powerful pitch pattern - I might be able to propose the same shape for small-scale student cloud-bill dashboards in our lab.

    • A ‘Corporate Memory’ / MemGraph pattern turns every incident into institutional knowledge, not just a closed ticket - a habit worth copying for our own group retros.

    • AWS-native Agentic platforms benefit enormously from running on Amazon Bedrock: the agentic tooling, the IAM story and the BYOC story all become simpler than running on a non-AWS LLM stack

  • Newly Acquired Skills

    • AgenticOps Mental Model: a working mental model of how specialised AI agents cooperate on a single operational loop - including the Super Agent orchestrator pattern and how the four sub-modules (Prevent, Resolve, Predict, Learn) interlock. Useful for any future agentic architecture I design.

    • Platform Architecture Reading: ability to read a vendor architecture diagram and translate it back into first principles (detection, decision, action, learning loop) - critical for evaluating any future AI-ops vendor or open-source project.

    • FinOps Intuition: concrete exposure to ROI-guaranteed FinOps models (Cloud Keepers), rightsizing heuristics and the math behind cloud-waste recovery (15-25%). Directly usable even on a student AWS account.

    • Compliance & Multi-region Awareness: familiarity with the parallel delivery of SOC 2, HIPAA, GDPR on a single global AI platform (Diaflow case) - even as a student this shapes how I write IAM and data-handling code in team projects.

    • AWS Bedrock Mental Model: a clearer sense of how Amazon Bedrock fits into a production agent stack (model selection, token-cost planning at ~$200 / user and BYOC vs. SaaS trade-offs). Shapes my own AWS certification and capstone choices.

    • Soft Skills: sharper listening during a sales-led platform showcase without losing analytical judgment; structured note-taking that separates platform promises, concrete numbers and case-study evidence; clearer questions about pricing, isolation model and student-friendly entry points.

Closing Note

CloudThinker’s session pulled the curtain back on what an actual enterprise AgenticOps platform looks like in 2026 - not a chatbot demo, but a four-module loop of specialised agents running on AWS Bedrock and shipped as both SaaS and BYOC. By pairing the four-module framework with concrete numbers (MTTR -87%, 70% auto-resolved, 15-25% waste recovery, toil cut from 40% to <10%, zero-downtime EKS upgrades) and three crisp case studies (F88, NextPay, Diaflow), the company turned an abstract ‘AI for cloud’ idea into something a student can reason about, compare and discuss. As a student I walked away with a reusable agentic-ops mental model, a sharper eye for FinOps and compliance claims, and a much clearer picture of where Amazon Bedrock-based platforms fit in the AWS ecosystem I am studying for my certifications.

2. Building Voice Agent at Scale - From real-time audio to production agent runtime

An AWS hands-on workshop hosted as part of the AWS First Cloud AI Journey (FCAIJ) program. The session walked students from a working voice agent demo (“Hera” - a voice-first Apple Store assistant) all the way to a production agent-runtime architecture on AWS, teaching both the moving parts and the design choices.

Speakers

  • Trung Vu, Nghi Danh, Kiet Tran – AWS community authors / builders.

Summary of content and main activities

The speakers opened with a workshop outcomes map: by the end, participants should be able to (1) choose the architecture - speech-to-speech vs cascaded STT-LLM-TTS and know where latency enters, (2) scale the agent brain with tools, sub-agents and session segmentation, and (3) ship the runtime using WebSocket, WebRTC, telephony, AgentCore Runtime and observability. They then anchored every concept in the running example of Hera - a voice-first Apple Store assistant that only needs a browser microphone.

  • Activity 1 — Voice Agents Are Streaming Systems

    • Framed the core principle: natural conversation needs simultaneous streams, not request/response turns. Mapped the channels: client to agent (browser, mobile, edge, phone), agent to model (bidirectional audio + tool events), telephony (SIP / voice provider handoff). Target: sub-second feel. Risk: jitter and stalls. Fix: stream each hop.

  • Activity 2 — Architecture Choice: Speech-to-Speech vs Cascaded

    • Compared unified speech-to-speech models (lowest moving-parts count, built-in streaming, turn-taking, voices, tool use) with cascaded STT-LLM-TTS pipelines (more control over each component and language coverage). Decision frame: optimize for natural flow first; choose modularity only where it buys real capability.

  • Activity 3 — Transport Choices (WebSocket, WebRTC, Telephony, KVS TURN)

    • Mapped the network paths - WebSocket for simple full-duplex prototypes, WebRTC + TURN for UDP-first low-latency media, Managed WebRTC for SFU + SDKs offload, Telephony for inbound/outbound calls, Amazon Kinesis Video Streams TURN for AWS-native NAT traversal, and SageMaker + vLLM STT for open real-time ASR models.

  • Activity 4 — Latency Is the Product

    • Highlighted the workshop rule: users judge intelligence by timing before they judge model quality. Focused on time-to-first-audio (silence after the user stops), barge-in (interrupting mid-response), and network recovery (reconnects as first-class UX paths).

  • Activity 5 — Build Path: Prototype to Production (4 Decisions)

    • Introduced the four decisions that turn a voice demo into a production architecture: Model (Nova Sonic or cascaded), Transport (WebSocket / WebRTC / telephony), Runtime (AgentCore, Fargate, SageMaker), and Operations (Evals, policy, cost, handoff).

  • Activity 6 - AgentCore Runtime Deployment Pattern

    • Detailed the production path: package the agent pipeline as a container (ARM64 / Graviton image), expose /ws or WebRTC entrypoints, isolate each session in an AgentCore Runtime microVM, use TURN/VPC design when WebRTC is required, and trace audio, reasoning and tool calls.

  • Activity 7 - Scaling the Agent Brain

    • Three patterns: tool calling (Nova Sonic invokes MCP tools through AgentCore Gateway for low-latency actions), agent-as-tool (delegate complex reasoning to specialized agents with their own prompts and tools), and session segmentation (split authentication, inquiry and fulfillment into focused voice sessions).

  • Activity 8 - Use Cases Worth Stealing

    • Reviewed real-world patterns inspired by AWS examples: healthcare contact center (Clarus Care - multi-intent voicebot and urgent routing), media assistant (movie recs + scene Q&A via AgentCore Gateway + RAG), podcast generator (Nova 2 Sonic for two-host audio), smart devices / vehicles (Nova Sonic + WebRTC) and high-volume support (S2S with structured handoff).

  • Activity 9 - Production Readiness Checklist

    • Exit criteria: p50/p95 time-to-first-audio tracked, barge-in + VAD tested, tool failures degrade gracefully, human handoff has context. Checklist axes: latency + audio, reliability (reconnects, retries, circuit breakers), trust + access (SigV4/OAuth, guardrails, PII boundaries), and cost + operations (session cleanup, evals, dashboards).

  • Activity 10 - Hera: Voice-First Apple Store Assistant

    • Introduced the running demo: a browser-only voice assistant for the Apple Store, powered by Bedrock + Nova 2 Sonic + S3 Vectors + Pipecat. Features: speech-to-speech with Nova 2 Sonic, RAG voice tool calling (“lookup_product”) and only a browser microphone required.

  • Activity 11 - Live Conversation Sample

    • Walked through a real Hera conversation. User: “Do you have MacBook Pro M4 in stock?” Agent answers with 14-inch M4 Pro pricing; user asks about iPhone 13 Pro Max and Hera reports out-of-stock + offers alternatives - all grounded in a 3-SKU knowledge base on S3 Vectors.

  • Activity 12 - Overall Architecture (Edge + Compute/AI Layer)

    • Reviewed the fully managed architecture: edge layer for browser WebSocket and audio capture, compute/AI layer for AgentCore Runtime + Bedrock + S3 Vectors + Lambda presigner. Emphasized the absence of self-managed ECS/EC2/VPC.

  • Activity 13 - Amazon Bedrock + Nova 2 Sonic Deep Dive

    • Explained why Nova 2 Sonic was chosen: ASR + LLM + TTS in a single stream, bidirectional WebSocket, built-in tool calling the model directly invokes, natural barge-in, and async tool calling so the agent keeps speaking while the tool runs. Latency under 2s with no need to glue three vendors together.

  • Activity 14 - Bedrock Knowledge Base + S3 Vectors

    • Mapped the RAG layer: catalog/*.md -> S3 source -> Bedrock ingestion job -> Titan v2 embeddings -> S3 Vectors. Retrieval pipeline: “lookup_product(query)” -> Retrieve API -> S3 Vectors search -> top-3 chunks -> Sonic reply. Why S3 Vectors vs OpenSearch Serverless: truly serverless (no OCU floor), native Bedrock integration, ideal for spiky read-light workloads.

  • Activity 15 - Voice Turn Flow (5 Steps, p95 < 3s)

    • Walked through a single turn: (1) browser requests presigned WSS URL (Lambda presigner, SigV4, 5-min TTL), (2) browser opens WebSocket to AgentCore (no SDK), (3) Pipecat receives 16 kHz mono Int16 PCM, (4) Sonic calls lookup_product -> Bedrock KB Retrieve -> S3 Vectors -> top-3 chunks, (5) Sonic outputs 24 kHz audio for browser playback via audio worklet. End-to-end < 3s p95.

  • Activity 16 - Real Metrics

    • Cost numbers: < $1 per hour of burst test, ~$0 idle overnight (AgentCore $0, S3 Vectors $0). Quality: KB top score 0.86 on iPhone 13 stock query. Latency: < 3s p95 end-of-utterance to audio. Pricing of Nova 2 Sonic: ~$0.017/min conversation.

  • Activity 17 - Recap and Workshop Closing

    • Drew the formula: 1 speech-to-speech model (Nova 2 Sonic) + tool calling + RAG on managed infrastructure (AgentCore, S3 Vectors). Emphasized Pipecat’s role in 1-day-to-run audio framing. Closing quote: ‘cheap, lean, observable - no self-managed ECS/EC2/VPC, ~$0 idle.’

  • Activity 18 - Q&A and Discussion

    • Asked follow-up questions on S3 Vectors pricing at scale, how to reproduce Hera setup on AWS Free Tier for student projects, and which parts of the workshop are reusable for a non-Apple voice-assistant use case (e.g. university admissions chatbot).

Results and values ​​achieved

  • Lessons learned

    • Voice agents are streaming products, not chatbots with a microphone - concurrent audio + tool + model streams are the delivery model, not an optimization.

    • Latency is the product - a ‘slower but smarter’ agent will always lose to a ‘slightly less smart but instant’ one. Time-to-first-audio is the KPI.

    • Use-case first, architecture second - the transport (WebSocket vs WebRTC vs telephony), memory boundary and tool surface are all determined by the use case, not by default tech preferences.

    • Managed infrastructure is a competitive advantage for small teams - AgentCore, S3 Vectors and Bedrock Knowledge Base eliminate 90% of the DevOps work required to host a real-time AI service.

    • Sub-agents and tool calling are the right abstractions to keep voice agents maintainable - one big prompt is the fastest way to ship an agent that breaks under growth.

  • Newly Acquired Skills

    • Real-Time AI Architecture: Ability to design an end-to-end audio path: browser -> WebSocket -> Pipecat -> Nova 2 Sonic -> Bedrock KB -> S3 Vectors, with isolation, presigning and tracing.

    • AWS Media / Streaming Stack: Familiarity with Bedrock AgentCore Runtime, AgentCore Gateway, Amazon Bedrock Knowledge Base, S3 Vectors, Kinesis Video Streams TURN and Lambda presigning for SigV4 WebSocket URLs.

    • Voice UX & Quality: Knowledge of how to optimize time-to-first-audio, design for barge-in, VAD, and graceful tool failure for natural conversation.

    • Cost & Operations Awareness: Practical understanding of how serverless scale-to-zero gives near-$0 idle cost and ~$1/hour burst cost - very valuable for budget-conscious student projects.

    • Soft Skills: Practiced focused listening during a fast, demo-heavy session, structured note-taking on architecture + metrics, formulating precise workshop-style questions, and peer discussion afterwards on replicating the design in coursework.

Closing Note

The ‘Building Voice Agent at Scale’ workshop was the moment I made the jump from ‘I can call an LLM’ to ‘I can ship a real-time AI product’. The Hera demo proved that with managed AWS services, a small team - even a student team - can build a real voice assistant that is fast, observable and nearly free at idle. As a student I now have a clear blueprint, a tighter architecture vocabulary, and a community to keep growing with.

3. AWS DevOps Agent: Your Always-Available Operations Teammate

A product/architecture session inside the First Cloud AI Journey community track, delivered by Ms. Bao and Mr. Nguyen Nguyen from Cloud Magnetic. The talk positioned AWS DevOps Agent as an autonomous AI teammate that drives operational excellence by resolving and proactively preventing incidents across AWS, Azure and on-premise environments. It opened with the real cost of manual incident investigation (high Mean Time To Detect, extended Mean Time To Resolve, context loss, reactive fire-fighting), then walked through the agent’s solution shape (triggered response, issue prevention, on-demand tasks), the six core pillars that make it different, the four-stage incident lifecycle (Triage - Investigation - Mitigation - Prevention) and finally the kinds of workloads where it fits best, plus three real-world use case numbers.

Speakers

  • Ms. Bao and Mr. Nguyen Nguyen – Cloud Magnetic.

Summary of content and main activities

Ms. Bao and Mr. Nguyen Nguyen opened the talk by grounding it in a familiar engineering reality: incident investigation is still mostly manual, and the manual version is expensive. They sketched the typical scenario - a customer trigger fires, telemetry is fragmented across multiple tools, knowledge gaps appear on the on-call engineer, and the engineer is constantly interrupted - and listed the resulting friction: high Mean Time To Detect (MTTD), extended Mean Time To Resolve (MTTR), context loss between shifts and reactive problem solving instead of proactive prevention. From there they introduced AWS DevOps Agent as an autonomous AI teammate that drives operational excellence by resolving incidents AND proactively preventing them across AWS, Azure and on-prem, then explained how it is built differently from a generic chatbot: six core pillars (Context Learning, Control, Integration, Collaboration, Convenient, Cost Effective) and a four-stage incident lifecycle (Triage, Investigation, Mitigation, Prevention) that always sits on top of an Application Topology Graph as the single source of truth. They closed with a clear ‘best fit’ guardrail - good observability, large-scale services and human-in-the-loop approval - and three quantified use cases showing 77% faster MTTR, 75% less time per incident, and approximately 73% faster incident response.

  • Activity 1 — Opening Scene: The Real Cost of Manual Incident Investigation

    • Set the stage with the typical incident scenario: a customer trigger arrives, telemetry is fragmented across tools, the on-call engineer faces knowledge gaps and is constantly interrupted. Showed the friction it produces - high MTTD, extended MTTR, context loss between handoffs and reactive problem solving - and made the case that ‘fewer incidents’ is not the only goal; faster, more informed response is the real lever.

  • Activity 2 — Introducing AWS DevOps Agent as an Operations Teammate

    • Positioned AWS DevOps Agent as an autonomous AI teammate, not a generic chatbot. Its core mission: drive operational excellence by resolving AND proactively preventing incidents across AWS, Azure and on-premise environments. Framed the agent with three working modes - Triggered Response when an incident fires, Issue Prevention as a continuous background activity and On-demand Tasks invoked by engineers.

  • Activity 3 — The Application Topology Graph (The Single Source of Truth)

    • Walked through the architectural core. The Application Topology Graph is the always-up-to-date map of services, dependencies, state and memories that the agent uses to actually understand what is going on. Without this graph the agent would be guessing; with it, every stage of the incident lifecycle is grounded in real structure rather than free-floating chat context.

  • Activity 4 — Core Pillar 1: Context Learning (Topology, State, Memories)

    • First differentiator. The agent continuously learns topology (which service talks to which), state (what is currently running and what just changed) and memories (what happened during past incidents and how they were resolved). This is what turns the agent from a stateless LLM into a long-lived teammate that knows your environment.

  • Activity 5 — Core Pillar 2: Control (Governed Control Spaces)

    • Second differentiator. The agent operates inside Governed Control Spaces - clearly defined boundaries for what it can read, what it can propose and what it can execute. This is the answer to the classic ‘AI is too risky in production’ objection: the blast radius is explicit and bounded.

  • Activity 6 - Core Pillar 3: Integration (MCP, AWS Support and more)

    • Third differentiator. Native integration with Model Context Protocol (MCP) servers, AWS Support APIs and the wider AWS ecosystem means the agent can fetch real data and call real actions rather than synthesise answers in isolation. Integrations are the bridge from ‘reads diagrams’ to ’touches your infrastructure’.

  • Activity 7 - Core Pillar 4: Collaboration (Slack, ServiceNow, PagerDuty)

    • Fourth differentiator. The agent lives where engineers already communicate: Slack channels, ServiceNow tickets, PagerDuty rotations. Incidents and proposals show up in the existing collaboration surfaces so engineers do not have to context-switch.

  • Activity 8 - Core Pillar 5: Convenient (Zero Password Setup)

    • Fifth differentiator. Zero-password setup means onboarding is friction-free: no separate credentials to manage, no separate portal to learn, the agent plugs into the existing identity and access story. Low-friction onboarding is what allows adoption to scale beyond a couple of champions.

  • Activity 9 - Core Pillar 6: Cost Effective (Pay-per-Second Billing)

    • Sixth differentiator. Pay-per-second billing means the agent’s cost scales with usage: when nothing is happening, you are not paying; during an incident, you pay for the seconds the agent actually works. This is the economic shape that makes it realistic even for smaller teams.

  • Activity 10 - Stage 1 of Incident Lifecycle: Triage (Correlate)

    • First lifecycle stage. Triage correlates incoming signals - alerts, customer reports, anomalies - into a single, prioritised view of ‘what is actually happening right now’. This collapses the MTTD problem into a single conversation rather than a multi-tool investigation.

  • Activity 11 - Stage 2 of Incident Lifecycle: Investigation (Analyze and Reason)

    • Second lifecycle stage. The agent analyses the correlated signals and reasons across the Application Topology Graph to identify the most likely root cause. This is where Context Learning pays off - past incidents, similar patterns and service dependencies all feed into a ranked list of hypotheses.

  • Activity 12 - Stage 3 of Incident Lifecycle: Mitigation (Plan and Remediation)

    • Third lifecycle stage. The agent proposes a remediation plan - what to change, where to change it and in what order - but does not auto-execute destructive actions without approval. The planner becomes a co-pilot for the engineer, not a black box.

  • Activity 13 - Stage 4 of Incident Lifecycle: Prevention (Postmortem and Action)

    • Fourth lifecycle stage. After mitigation, the agent writes the postmortem and proposes follow-up actions so the same class of incident is less likely to happen again. This closes the loop from reactive fire-fighting to proactive prevention.

  • Activity 14 - Best Fit 1: Good Observability (CloudWatch Data)

    • First ‘best fit’ guardrail. The agent is most useful when it has clean observability data from CloudWatch - logs, metrics, traces, alarms, deployment history and past incident data. Without this input the agent has to guess; with it, it can reason properly.

  • Activity 15 - Best Fit 2: Large-Scale Services

    • Second ‘best fit’ guardrail. The agent’s recommendations become more focused and useful at large-scale services where humans simply cannot hold all the moving parts in their head. Small toy services rarely need this level of automation; production-scale services do.

  • Activity 16 - Best Fit 3: Recommends Only (Human-in-the-Loop)

    • Third ‘best fit’ guardrail. The agent recommends next best steps but a human remains the final approver and executor. This is the governance answer: the agent accelerates decisions, humans still own them.

  • Activity 17 - Use Case Outcome 1: 77% Faster MTTR

    • First real-world quantification. Result reported by the speakers: 77% faster Mean Time To Resolve on representative incident types. The reduction comes mainly from collapsing the Investigation stage and making Mitigation more accurate.

  • Activity 18 - Use Case Outcome 2: 75% Less Time per Incident

    • Second real-world quantification. 75% reduction in the time an engineer spends on a single incident, end-to-end. Most of the saved time is in repetitive steps - correlating logs, opening tickets, writing the initial summary - that the agent can now do in seconds.

4. Building Secure Private MCP for Quick (Amazon Quick / Amazon Q Business)

A security-and-architecture session inside the First Cloud AI Journey community track, co-presented by Toan Nguyen (AWS Security Builder) and Nguyen Hoang Danh Nghi (Renova Cloud). The talk unpacked how Amazon Q supports the Model Context Protocol (MCP) connector by default but routes over the public Internet to a public MCP endpoint by default - which produces a real list of enterprise blockers (forced public endpoint, insecure transport for Context and logs and broken Data Residency and Zero Trust policies). It then walked the audience through VPC Connection for Quick as the secure replacement (Quick ENI on a private IP plus Amazon Route 53 Inbound Endpoint plus Private ALB plus Internal Service on ECS or EC2) and finally spelled out the five-step request flow with TLS 443 termination at the Internal ALB and Jaeger as the centralised tracing backend.

Speakers

  • Toan Nguyen (AWS Security Builder) and Nguyen Hoang Danh Nghi (Renova Cloud)

Summary of content and main activities

Toan Nguyen and Nguyen Hoang Danh Nghi opened with the working console of Amazon Quick: the ‘Welcome to Quick Suite’ hub with Activities, Dashboards, Automate, Datasources and Flows on the left and an intentional Chat prompt - ‘Good evening! Let’s chat’ - that forces the user to pick a Space before asking anything. From that entry point they moved to MCP architecture (Host with MCP Client talking to many specialised MCP Servers each guarding its own data domain) and then to the uncomfortable default: although Amazon Q supports the MCP connector out of the box, it routes over the public Internet to a public endpoint - which creates three structural blockers (forced public endpoint with increased attack surface, insecure transport for Context and logs, and broken Data Residency plus Zero Trust plus Internal-only policies). The core of the talk was the secure replacement: VPC Connection for Quick, with Quick ENI on a private IP, Amazon Route 53 Inbound Endpoint, Private ALB and Internal Service on ECS or EC2 - anchored on three principles (Private entry, Private DNS and Internal path). They closed by walking the audience through a concrete five-step request flow and pointing out the operational benefits (stable hostnames despite rolling deploys), compliance benefits (Data Residency + Zero Trust) and three explicit anti-patterns (public endpoints for convenience; relying on IP allow-lists instead of Private DNS; TLS termination before entering customer trust zone).

  • Activity 1 — Welcome to Amazon Quick Suite: The Agent Builder Interface

    • Walked through the working console. The ‘Welcome to Quick Suite’ hub lets users spin up specialised Chat agents or use the regular Chat to interact directly with the enterprise knowledge base. The left navigation exposes Activities, Dashboards, Automate, Datasources and Flows. The right Chat window greets the user with ‘Good evening! Let’s chat’ and lets them select a Space before asking anything - an intentional guard-rail at the entry of every prompt.

  • Activity 2 — MCP Architecture: Host, Protocol Layers and Endpoint Servers

    • Mapped the architectural foundation. A Host (e.g. Claude Desktop, IDE, Tools) runs an MCP Client that connects over multiple parallel protocol streams - MCP Protocol, SSE Protocol and MCP Protocol again - down to dedicated MCP Servers. Each server speaks to its own data source: MCP Server A to Local Data Source A, MCP Server B to Local Data Source B, MCP Server C through a Web API gateway to Remote Service C. The model is one client, many specialised servers - a clean blast-radius boundary.

  • Activity 3 — The Context: Risks of Amazon Q Default Public MCP Connector

    • Set the security backdrop. Amazon Q supports the MCP Server Connector by default, but the default path runs over the Public Internet straight to a Public MCP Endpoint. This produces three structural risks: Required public endpoint (increases attack surface), insecure transport for Context and logs (interception risk in transit) and policy violations against Data Residency, Zero Trust and Internal-only policies.

  • Activity 4 — Solution: VPC Connection for Quick as the Private Backbone

    • Introduced the secure architecture that replaces the public path. The end-to-end flow stays inside a private VPC (10.0.0.0/16): Amazon Quick connects via a Quick ENI on a private IP through Amazon Route 53 Inbound Endpoint to a Private Application Load Balancer and from there to Internal Service on ECS or EC2 hosting the MCP Server. Three core principles underline this design: Private entry (no public endpoint ever), Private DNS (hostnames only resolve inside the VPC) and Internal path (every hop is a private IP).

  • Activity 5 — Step 1 of Request Flow: Quick Sends Query via VPC Connection

    • Walked through the five-step request flow. Step 1: Quick dispatches the MCP query through its dedicated VPC Connection into the customer-owned network rather than over the public Internet.

  • Activity 6 - Step 2 of Request Flow: ENI Looks Up ALB via Route 53 Resolver

    • Step 2: The Quick-side Elastic Network Interfaces issue a DNS query to Route 53 Resolver to obtain the private IP addresses of the Internal ALB - keeping the entire lookup inside VPC DNS rather than leaking names out to public resolvers.

  • Activity 7 - Step 3 of Request Flow: HTTPS 443 with TLS Termination at ALB

    • Step 3: The request travels over encrypted HTTPS on port 443 to the Internal ALB. TLS terminates at this internal load balancer so that session security is guaranteed end-to-end from Quick all the way into the customer network without ever leaking plaintext across public hops.

  • Activity 8 - Step 4 of Request Flow: ALB Forwards via HTTP 8000 to MCP Server

    • Step 4: Once TLS is terminated inside the trusted perimeter, the ALB forwards the request over plain HTTP on port 8000 to the MCP Server, operating inside its own internal security zone where eavesdropping is mitigated by network isolation itself.

  • Activity 9 - Step 5 of Request Flow: MCP Server Queries Jaeger and Returns Result

    • Step 5: The MCP Server queries Jaeger for distributed-tracing data, aggregates everything that was asked for and returns the final response back through the same private path to the user.

  • Activity 10 - Why Private DNS Matters in This Design

    • Reflection on Principle 2 (Private DNS). Because hostnames are only resolvable inside the VPC, even if an attacker learns a target alias name from logs or documents they cannot resolve it from outside. This is what makes ‘internal-only’ actually enforceable rather than just being a wish on paper.

  • Activity 11 - Why Internal Path Matters at Every Hop

    • Reflection on Principle 3 (Internal path). Every hop ENI to ALB to MCP is a private IP. Compared with a hybrid public private design, this removes any chance of a transient leak from a misconfigured NAT gateway or a forgotten security group rule. Single-direction, single-trust-zone.

  • Activity 12 - Mapping Default Risks to VPC Mitigations

    • Pulled together how each default risk is neutralised by the new design. Required public endpoint is replaced by Private entry and Quick ENI; insecure transport is replaced by TLS at HTTPS 443 terminating at Internal ALB; Data Residency plus Internal-only policy violations are replaced by Private DNS plus Internal path - every concern has a concrete architectural answer.

  • Activity 13 - Why This Pattern Works for Sensitive Workloads

    • Showed which workloads benefit most. Anything involving PII, internal HR data, financial ledgers or proprietary source code fits this pattern best because it gives them enterprise-grade network isolation by default without changing how engineers interact with Amazon Quick.

  • Activity 14 - Why This Pattern Works for Multi-Cloud or Hybrid Architectures

    • Showed cross-cloud relevance. The pattern of one host client many private MCP servers works equally well when each server connects to a different data domain - one talks only to PostgreSQL on RDS, one only to GitHub Enterprise, one only to an internal SLA API. That specialisation at server level mirrors specialisation at team level.

  • Activity 15 - Operational Benefit: Stable Hostnames Plus Rolling Deployments

    • Noted an operational benefit. Because Route 53 inbound endpoints sit in front of ALBs that sit in front of ECS tasks, the customer can do rolling deployments or even blue-green on their MCP servers without changing any hostname that Amazon Quick was configured with.

  • Activity 16 - Operational Benefit: Centralised Observability via Jaeger

    • Noted the observability lever. By pointing every MCP server integration at Jaeger as its tracing backend (as shown in Step 5), operators get one pane for end-to-end latency, error budgets and per-tool dependency health rather than scattered traces across each service.

  • Activity 17 - Compliance Benefit: Data Residency plus Zero Trust Coverage

    • Compliance framing. Data Residency requirements normally need contractual proofs plus operational proofs; here operational proof comes from ‘data never leaves VPC’. Zero Trust coverage comes from explicit allow-listed network paths instead of allowing VPNs broad reach once joined.

  • Activity 18 - Anti-Pattern Warning: Public Endpoints Just for Convenience

    • Warning slide. Treating public MCP endpoints as acceptable because they are ‘just internal APIs’ defeats the whole architecture - every public hop is also an attack surface and an audit gap. Convenience should never win over blast-radius reduction here.

  • Activity 19 - Anti-Pattern Warning: Skip Private DNS and Rely Only on IP Allowlists

    • Warning slide against relying purely on IP allow-listing without Private DNS. IP allow-lists drift; Private DNS rooted inside Route 53 gives both readable names AND stable addresses and stays correct across redeploys.

  • Activity 20 - Anti-Pattern Warning: TLS Termination Outside Customer Trust Zone

    • Warning slide against TLS termination before reaching customer-controlled infrastructure. If TLS terminates at an Internet-facing LB before entering customer trust zone, sensitive Context payloads briefly become accessible in plaintext - defeating principle of least exposure.

  • Activity 21 - Speaker Takeaway Toan Nguyen: Security Builder Mental Model

    • Toan Nguyen’s framing as AWS Security Builder emphasised treating agent infrastructure like any other production workload - same VPC isolation patterns, same TLS expectations, same observability bar. Nothing about an ‘agent’ changes what counts as secure.

  • Activity 22 - Speaker Takeaway Danh Nghi: Renova Cloud Delivery Lens

    • Nguyen Hoang Danh Nghi from Renova Cloud emphasised delivery practicality - what customers actually need on day one is a repeatable pointer pattern that works across existing AWS accounts, hence Route 53 inbound endpoint plus Internal ALB plus ECS or EC2 standing up as the canonical answer.

  • Activity 24 - Student Mapping: How I Could Replicate This in Our Lab Project

    • Student-level mapping. Even at hobby scale I could adopt a slimmed-down version of this pattern: deploy our group’s small chatbot inside a private subnet behind an ALB; rely on internal DNS names; keep ingestion strictly internal even for a toy project so we train on the right habits early.

  • Activity 25 - Closing Synthesis: Private-by-Default Is an Architectural Choice You Make Once

    • Closing synthesis. Building secure private MCP for Quick is not about adding more security controls; it is about choosing once that agents will live in private network planes by default. Once that choice is made every downstream decision - which connector, which LB, which DNS zone - follows naturally from it.

Results and values ​​achieved

  • Lessons learned

    • The default public-Internet MCP path is not ‘internal because it looks internal’; it is structurally a public endpoint and that matters for Data Residency, Zero Trust and audit conversations. Treat default AI connectors with the same scepticism you would treat any default database port exposed to 0.0.0.0/0.

    • Three principles cover 90 percent of the secure design: Private entry (no public endpoint ever), Private DNS (hostnames only resolve inside the VPC) and Internal path (every hop is a private IP). If your proposal cannot defend all three on paper, it is not actually a private architecture.

    • TLS termination at the Internal ALB is the keystone of trust. Terminating earlier (for example at an Internet-facing LB) briefly exposes sensitive Context as plaintext; terminating later than the Internal ALB is fine; terminating exactly at the Internal ALB is the design sweet spot.

    • Specialised MCP servers protect blast radius. Putting Postgres, GitHub and an internal SLA API behind separate servers means a compromise in one cannot sweep down all integrations - this is the agent-world equivalent of least-privilege IAM.

    • Centralised tracing via Jaeger turns observability from ‘good luck’ into ‘one pane’. Because every MCP call ends in a Jaeger span, latency budgets and per-tool error rates become a single dashboard rather than three separate logs.

    • Anti-patterns matter as much as patterns. The session’s warnings about public endpoints, IP-allowlist-only designs and TLS termination outside trust zones are worth memorising because they are how security work quietly regresses in agent projects under time pressure.

    • HR Workflow Reading: ability to map a real HR day (connect data sources, analyse candidates, generate reports, collaborate with managers, schedule interviews and offers, update pipeline tracking) onto AI-actionabMCP Architecture Reading: ability to look at any new MCP connector offered by a vendor and answer three questions immediately - is the endpoint public; is transport encrypted; and what identity model does it use to talk back to my data sources.

    • Private-VPC Connectivity Mental Model: a working map of Quick ENI plus Route 53 Inbound Endpoint plus Internal ALB plus ECS or EC2 - enough vocabulary to sketch the architecture on a whiteboard and defend each piece against a security review.

    • Five-Step Request Flow Pattern: the ability to draw a numbered call sequence from ‘agent request’ through ENI DNS lookup through TLS termination through ALB forward to MCP server and back, and to spot where hop-level risks could be inserted if someone changes a link.

    • Jaeger-Style Tracing Pattern: practical knowledge of using a tracing backend as the single source of truth for AI tool latency / errors / dependencies - every MCP server need only emit one span to be visible in the same pane.

    • Data-Residency Vocabulary: concrete vocabulary for explaining to product and security teams why ‘internal-only’ must be enforced via Private DNS rather than via documentation promises - a useful habit for any future agent project that touches PII or financial data.

This session quietly re-classified what ‘secure MCP for an AI agent’ actually means at the network layer. The argument was not ‘MCP is unsafe, do less of it’, but ‘MCP is unsafe by default, route it through a VPC Connection for Quick and the design becomes safe by construction’. By the end of the talk I had a vocabulary my future projects can lean on: Quick ENI plus Route 53 plus Internal ALB plus MCP server; three principles (Private entry, Private DNS, Internal path); five explicit request-flow steps; centralised observability via Jaeger; and a short anti-pattern list that catches the most common regressions. The clearest shift was from thinking of MCP servers as ‘just endpoints’ to thinking of them as specialised, isolated tools each guarding one data domain, and of agent traffic as something that must stay inside the customer trust zone from request to response. That frame - private-by-default as an architectural choice you make once - is what I will carry forward to every future agent project.
